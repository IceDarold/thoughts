Урок 3.1. Интуиция и математика user-based CF

Цели

  • Объяснить идею user-based CF на пальцах.
  • Разобрать базовую формулу предсказания рейтинга/скора.
  • Показать, как из матрицы «user–item» извлекаем «похожих пользователей».

Теория

  1. Интуиция user-based CF
    • Если два пользователя ставили похожие оценки/совершали похожие действия:
      • у них похожие предпочтения.
    • Чтобы предсказать, понравится ли item пользователю u:
      • ищем «соседей» u (похожих пользователей);
      • смотрим, как они относятся к item;
      • усредняем их мнения с весами по похожести.
  1. Матрица взаимодействий
    • R[u, i] — рейтинг или сигнал взаимодействия (может быть 0/1).
    • Очень разреженная матрица.
    • Обозначения:
      • N(u) — множество пользователей, похожих на u;
      • I(u) — множество items, с которыми взаимодействовал u.
  1. Similarity между пользователями
    • Косинусное сходство:

      [

      sim(u, v) = \frac{\sum_i R[u, i] R[v, i]}{\sqrt{\sum_i R[u, i]^2} \sqrt{\sum_i R[v, i]^2}}

      ]

      суммирование по общим item’ам.

    • Pearson корреляция (для рейтингов):
      • вычитаем средний рейтинг каждого пользователя.
  1. Предсказание рейтинга: простой вариант
    • Без учёта смещений:

      [

      \hat{r}{u, i} = \frac{ \sum{v \in N_k(u)} sim(u, v) \cdot R[v, i] }{ \sum_{v \in N_k(u)} |sim(u, v)| }

      ]

      где N_k(u) — k ближайших соседей пользователя u.

    • С учётом средних:
      • для рейтингов: используем отклонения от среднего рейтинга пользователя.
  1. Top-K рекомендации из user-based CF
    • Для пользователя u:
      • собрать кандидатов — все items, которые выбирали его соседи N_k(u), но не выбирал он сам;
      • оценить \hat{r}_{u, i} для этих items;
      • отсортировать и взять топ-K.
  1. Ограничения user-based CF
    • Требует относительно «плотных» профилей:
      • если у пользователя мало взаимодействий, соседей найти сложно;
    • При большом числе пользователей дорого хранить/считать sim(u, v).

Практика на занятии

  • В ноутбуке:

      • выбрать подмножество пользователей и items (например, 5k × 5k) для учебных целей;
    1. Реализовать:
      • функцию для подсчёта cosine similarity между двумя пользователями (по общим items);

    1. Для 2–3 пользователей:

      • интерпретировать: действительно ли их поведение похоже.

Урок 3.2. Item-based CF и выбор метрик сходства

Цели

  • Показать, почему в реальности чаще используют item-based CF.
  • Разобрать разные similarity-метрики.
  • Реализовать item-based CF и сравнить с user-based.

Теория

  1. Интуиция item-based CF
    • Если два предмета часто выбирают одни и те же пользователи, они похожи.
    • Для рекомендации пользователю u:
      • смотрим на items, с которыми он взаимодействовал;
      • для каждого — находим похожие items;
      • объединяем кандидатов и сортируем.
  1. Почему item-based часто практичнее
    • Пользователей обычно больше, чем предметов (или наоборот, но чаще так).
    • Каталог предметов меняется медленнее, чем аудитория:
      • проще заранее посчитать «item → похожие items» и кешировать;
      • при разработке: проще хранить матрицу сходств по items.
  1. Меры сходства для items
    • Косинус:
      • вектора — строки/столбцы матрицы user–item.
    • Jaccard (для бинарных взаимодействий):

      [

      J(i, j) = \frac{|U(i) \cap U(j)|}{|U(i) \cup U(j)|}

      ]

      где U(i) — множество пользователей, взаимодействовавших с item i.

    • Adjusted cosine:
      • учитываем средние рейтинги пользователей (для explicit).
  1. Предсказание рейтинга/скора по item-based CF
    • Аналогично user-based:

      [

      \hat{r}{u, i} = \frac{\sum{j \in I(u)} sim(i, j) \cdot R[u, j]}{\sum_{j \in I(u)} |sim(i, j)|}

      ]

      где I(u) — items, с которыми взаимодействовал пользователь u.

  1. Построение топ-K рекомендаций
    • Для пользователя:
      • берём его I(u) (историю);
      • для каждого item j ∈ I(u) — находим ближайшие items;
      • агрегируем кандидатов, суммируя/максимизируя sim(i, j) или предсказанный score;
      • фильтруем те, что уже были;
      • берём топ-K.
  1. Item-based vs user-based: сравнение
    • Item-based:
      • проще кешировать;
      • часто более устойчив к появлению новых пользователей;
    • User-based:
      • интуитивно понятен;
      • иногда полезен, если профиль каждого пользователя очень плотный.

Практика на занятии

  • В ноутбуке:
    1. Построить матрицу item × item сходств (на подвыборке):
      • cosine similarity;
      • Jaccard similarity (если данные бинарные).
    1. Для нескольких items:
      • вывести топ-10 похожих items;
      • сравнить cosine vs Jaccard — какие соседи меняются, что кажется более адекватным.
    1. Реализовать item-based CF-рекомендации:

    # 1) взять историю I(u)
    # 2) для каждого из I(u) собрать похожие items
    # 3) агрегировать score
    # 4) вернуть топ-K

Урок 3.3. CF для implicit feedback и топ-K рекомендаций

Цели

  • Перейти от «рейтингов» к implicit feedback (клики, покупки).
  • Показать, что CF можно использовать не для регрессии, а сразу для top-K ranking.
  • Подружить CF-модели с eval-фреймворком из модуля 2.

Теория

  1. Explicit vs implicit в контексте CF
    • Explicit:
      • рейтинги 1–5, лайки/дизлайки;
    • Implicit:
      • просмотр, клик, покупка, добавление в корзину и т.п., часто без оценок;
    • Как кодировать implicit:
      • бинарно (0/1);
      • взвешенно (view < click < cart < purchase).
  1. CF для implicit: основная идея
    • Матрица R[u, i] = «есть взаимодействие» или «есть взвешенное взаимодействие».
    • Сходство считаем по таким векторам.
    • Предсказанный score = «как сильно мы ожидаем взаимодействия».
  1. Top-K напрямую
    • Не обязательно предсказывать точный рейтинг.
    • Можно сразу:
      • считать similarity;
      • генерить кандидатов;
      • сортировать по суммарному весу.
    • Метрики:
      • HitRate@K, Recall@K, MAP@K, NDCG@K — всё из модуля 2.
  1. Схема работы в implicit CF
    • Обучение = просто подсчёт статистик (similarities).
    • Для каждого пользователя:
      • история I(u) → кандидаты через ближайшие items/users;
      • score = агрегированный similarity/вес;
      • top-K → eval с теми же метриками, что и соревовая.
  1. Тонкости
    • Сильно популярные items:
      • тянут similarity вверх (если не нормировать);
      • могут доминировать.
    • Нужно:
      • нормировки;
      • отсечение слишком популярных items;
      • комбинирование с популярностью/recency (из модуля 1).

    1. Пересчитать similarity (item-based или user-based) по этой матрице.
    1. Реализовать:

        # 2) собираем кандидатов через похожие items/users

Урок 3.4. Инженерия CF, масштабирование и сравнение с бейзлайнами

Цели

  • Показать, как CF живёт в реальном мире: память, скорость, препроцессинг.
  • Обсудить масштабирование и приближённые методы.
  • Научить сравнивать CF-модели с простыми бейзлайнами и делать выводы.

Теория

  1. Сложность и память
    • Если мы наивно считаем similarity для всех пар пользователей/items:
      • O(N^2 * d) — ужас;
    • Нам обычно нужны только топ-N соседей:
      • заранее храним только k лучших соседей для каждого пользователя/item.
    • Sparse представление:
      • матрицы храним в CSR/CSC;
      • считаем similarity только для пар с пересечением.
  1. Pre-computation и кеширование
    • Precompute:
      • similarity между items (item-based) → сохраняем top-N соседи.
    • В онлайне:
      • для каждого пользователя строим рекомендацию за счёт уже посчитанного sim(i, j).
    • Трюки:
      • ограничивать число «соседей»;
      • выкидывать супер-редкие items;
      • выкидывать супер-популярные items из similarity.
  1. Approximate methods (на будущее)
    • ANN-поиск (FAISS и т.п.) — упоминание, без деталей:
      • можно хранить эмбеддинги и искать ближайших соседей быстро;
    • В курс можем дать это как «тизер» к более продвинутым модулям.
  1. Комбинирование CF с прошлым опытом (модуль 1)
    • CF + популярность:
      • когда у пользователя мало истории → fallback к сегментной популярности;
    • CF + co-occurrence:
      • co-occurrence можно использовать как быстрый генератор кандидатов;
      • CF — как более точный ranker.
    • CF + «бизнес-правила»:
      • фильтрация по наличию, цене, категориям и т.п.
  1. Анализ результатов
    • Сравнение по метрикам (из модуля 2):
      • Recall@K, MAP@K, NDCG@K.
    • Субъективная проверка:
      • для нескольких пользователей смотреть топ-K списки разных моделей;
      • оценивать: есть ли диверсификация, нет ли спама одним item’ом.
