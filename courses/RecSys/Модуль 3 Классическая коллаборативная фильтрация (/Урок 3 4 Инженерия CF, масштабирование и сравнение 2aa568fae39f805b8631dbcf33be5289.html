4. Сложность и память в CF

4.1. Наивная сложность

Разбор на уровне оценок:

  • Пусть:
    • U — число пользователей;
    • I — число items;
    • d — средняя длина вектора (кол-во ненулей).
  • Наивный пересчёт сходства:
    • user-user:
      • O(U² * d) — полный кошмар при U ≥ 10⁵;
    • item-item:
      • O(I² * d) — терпимее, но всё равно тяжело при большом I.
  • Хранение полной матрицы similarity:
    • user-user: O(U²),
    • item-item: O(I²),
    • нереально для больших.

Вывод: нам нужны приближения и ограничение количества соседей.

4.2. Разреженность и sparse-структуры

  • В реальных данных:
    • матрица R очень sparse, density << 1%;
  • Хранить её dense — дорого и бессмысленно.
  • Используем:
    • разреженные форматы (CSR/CSC) — scipy.sparse;
    • матричные операции → это сильно ускоряет.

Идея для урока:

  • показать разницу:
    • dense матрица 10k × 10k → много памяти;
    • sparse (CSR) → значительно меньше.

5. Инженерные приёмы: как не умереть на CF

5.1. Top-N neighbors и обрезка

Мы не храним similarity между всеми парами:

  • Для каждого item/user:
    • считаем сходство только с ограниченным числом «кандидатов»;
    • сохраняем только топ-N (например, N=100).

Практика:

  • item_neighbors[i] — список длины до N: (sim, j_idx).
  • Хранить такие списки гораздо дешевле, чем полную матрицу I×I.

5.2. Ограничение по популярности и частоте

CF страдает от:

  • суперпопулярных items («все любят i» → i похож на всех);
  • суперредких items (почти нет данных).

Приёмы:

  • выкинуть items:
    • слишком редкие (меньше N взаимодействий);
    • иногда — слишком частые (top 0.1–1% по популярности) из similarity;
  • ограничивать:
    • при расчёте соседей:
      • не считать similarity для пар с очень маленьким пересечением (|U(i) ∩ U(j)| < threshold).

5.3. Precomputation и кеширование

Обычно архитектура item-based CF в проде выглядит так:

  1. Offline (batch):
    • считаем item_neighbors для всех/части каталога;
    • сохраняем в базу / файл / кеш.
  1. Online (при запросах):
    • берем историю пользователя I(u);
    • берём соседей для каждого item из I(u) из кеша;
    • объединяем, ранжируем, отдаём top-K.

На уроке:

  • подчеркнуть разгрузку по time:
    • дорогое — offline;
    • дешёвое (инференс) — online.

6. Гибридные модели: CF + популярность / co-occurrence

6.1. Зачем гибрид?

CF не идеален:

  • плохо работает на холодных пользователях;
  • может странно себя вести на редких items.

Популярность и co-occurrence:

  • простые;
  • устойчивые;
  • хорошо работают при малых данных.

Идея:

«Давай дадим CF рулить там, где у нас много истории,

и fallback-иться на популярность там, где данных мало».

6.2. Схемы гибридизации

  1. Switching (переключатель):
    • Если у пользователя мало истории (например, < N событий):
      • используем чистую популярность / co-occurrence.
    • Если истории достаточно:
      • используем CF.
  1. Blending (смешивание скоров):
    • Для каждого кандидата считаем:
      • score_CF(u, i);
      • score_pop(i) (нормированную популярность);
    • Итоговый score:

      [

      score(u, i) = \alpha \cdot score_{CF}(u, i) + (1 - \alpha) \cdot score_{pop}(i)

      ]

    • α — гиперпараметр, настраиваемый по offline-метрикам.
  1. Multi-stage:
    • Сначала популярность/co-occurrence → кандидаты;
    • потом CF → rerank.

Для урока достаточно switching + blending.

7. Кодовая часть: оптимизированный item-based CF + HybridRecommender

7.1. Пример структуры HybridRecommender

class PopularityRecommender:
    def __init__(self, item_popularity, item_idx_to_id):
        self.item_popularity = item_popularity  # dict item_idx -> score
        self.item_idx_to_id = item_idx_to_id

    def recommend(self, user_id, k=20):
        # просто топ-k по популярности
        top_items = sorted(
            self.item_popularity.items(),
            key=lambda x: x[1],
            reverse=True
        )[:k]
        return [self.item_idx_to_id[i_idx] for i_idx, score in top_items]

class HybridRecommender:
    def __init__(self, cf_model, pop_model, min_history=3, alpha=0.7):
        self.cf_model = cf_model
        self.pop_model = pop_model
        self.min_history = min_history
        self.alpha = alpha

    def recommend(self, user_id, k=20):
        # 1) смотрим, сколько истории у пользователя
        user_idx = self.cf_model.user_id_to_idx.get(user_id)
        if user_idx is None:
            # полный cold start -> чистая популярность
            return self.pop_model.recommend(user_id, k)

        user_vec = self.cf_model.R[user_idx]
        history_len = int((user_vec > 0).sum())

        if history_len < self.min_history:
            # мало данных -> популярность
            return self.pop_model.recommend(user_id, k)

        # 2) есть история -> смешиваем CF и популярность
        cf_items = self.cf_model.recommend(user_id, k=3*k)  # берём запас
        pop_items = self.pop_model.recommend(user_id, k=3*k)

        # конвертируем в item_id -> score
        cf_scores = {item_id: (3*k - rank) for rank, item_id in enumerate(cf_items)}
        pop_scores = {item_id: (3*k - rank) for rank, item_id in enumerate(pop_items)}

        candidate_ids = set(cf_scores.keys()) | set(pop_scores.keys())

        blended = []
        for item_id in candidate_ids:
            s_cf = cf_scores.get(item_id, 0.0)
            s_pop = pop_scores.get(item_id, 0.0)
            score = self.alpha * s_cf + (1 - self.alpha) * s_pop
            blended.append((score, item_id))

        blended.sort(reverse=True, key=lambda x: x[0])
        return [item_id for score, item_id in blended[:k]]
