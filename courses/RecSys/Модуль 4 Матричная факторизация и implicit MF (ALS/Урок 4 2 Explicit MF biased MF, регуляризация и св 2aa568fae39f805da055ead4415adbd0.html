Урок 4.2. Explicit MF: biased MF, регуляризация и связь с топ-K

1. Цели урока

После урока студент:

  • понимает, зачем нужны:
    • глобальное среднее μ,
    • user-bias b_u,
    • item-bias b_i;
  • умеет записать модель:

    [

    \hat{r}_{u,i} = μ + b_u + b_i + \langle p_u, q_i \rangle

    ]

  • понимает функцию потерь biased MF и роль регуляризации:

    [

    L = \sum_{(u,i)\in \Omega} (r_{u,i} - \hat{r}_{u,i})^2 + \lambda(|p_u|^2 + |q_i|^2 + b_u^2 + b_i^2)

    ]

  • умеет реализовать biased MF на SGD (доработать SimpleMF из 4.1);
  • умеет:
    • считать RMSE для ratings;
    • посчитать хотя бы одну top-K метрику (например, Recall@K или NDCG@K) поверх предсказанных рейтингов;
  • понимает, почему минимизация RMSE ≠ оптимизация ранжирования, и как эти вещи могут расходиться.

2. Предварительные знания

Студенты уже умеют:

4. Теория

4.1. Почему простая MF без bias — не всегда ок

Проблема: пользователи и товары сильно различаются по «базовости».

  • Некоторые пользователи:
    • «добренькие» — почти всем ставят 4–5;
    • «злые» — почти всегда 1–3.
  • Некоторые фильмы/товары:
    • «универсально любимые» → даже требовательные ставят им 4–5;
    • «проходные» → даже лояльные ставят 2–3.

Если игнорировать это и всё объяснять только скалярным произведением p_u^T q_i:

  • факторы будут пытаться одновременно моделировать:
    • вкус пользователя,
    • «общую классность» item’а,
    • общую смещённость рейтингов.

Лучше разделить:

  • глобальный уровень рейтингов (μ),
  • «насколько этот пользователь в среднем выше/ниже среднего» (b_u),
  • «насколько этот item в среднем выше/ниже среднего» (b_i),
  • и уже остаток объяснять латентными факторами.

4.2. Biased MF: модель

Вводим:

  • μ — глобальное среднее всех рейтингов в train:

    [

    μ = \frac{1}{|\Omega|} \sum_{(u,i)\in\Omega} r_{u,i}

    ]

  • b_u — user-bias: насколько пользователь u обычно выше/ниже μ;
  • b_i — item-bias: насколько item i обычно выше/ниже μ.

Модель:

[

\hat{r}_{u,i} = μ + b_u + b_i + \langle p_u, q_i \rangle

]

Интерпретация:

  • μ — «средний рейтинг вообще»;
  • b_u — поправка «этот пользователь в среднем +0.3 к среднему»;
  • b_i — «этот фильм в среднем +0.5 к среднему»;
  • p_u^T q_i — «насколько именно этому пользователю именно этот фильм заходит сверх общей тенденции».

4.3. Функция потерь

Берём explicit MF и добавляем biases:

[

L = \sum_{(u,i)\in\Omega} (r_{u,i} - μ - b_u - b_i - \langle p_u, q_i \rangle)^2 + \lambda \left( \sum_u |p_u|^2 + \sum_i |q_i|^2 + \sum_u b_u^2 + \sum_i b_i^2 \right)

]

  • Первая часть: MSE на известных рейтингах.
  • Вторая часть: регуляризация:
    • penalize большие p_u, q_i (сложность латентного пространства),
    • penalize большие b_u, b_i (смещения тоже не должны раздуваться бесконечно).

Интуиция регуляризации:

  • без неё модель может подогнаться под шум:
    • каждому пользователю/товару дать «дико специфичный» bias/факторы;
  • с ней модель предпочитает более «простое» объяснение:
    • ближе к нулю факторы/смещения, если нет сильного сигнала.

4.4. SGD-обновления для biased MF

Для одной пары (u, i, r_ui):

  1. Предсказание:

    [

    \hat{r}_{u,i} = μ + b_u + b_i + \langle p_u, q_i \rangle

    ]

  1. Ошибка:

    [

    e_{u,i} = r_{u,i} - \hat{r}_{u,i}

    ]

  1. Обновления (learning rate — lr):
    b_u ← b_u + lr * (e_ui - λ * b_u)
b_i ← b_i + lr * (e_ui - λ * b_i)

p_u ← p_u + lr * (e_ui * q_i - λ * p_u)
q_i ← q_i + lr * (e_ui * p_u - λ * q_i)
  • e_ui тянет параметры в сторону уменьшения ошибки.
  • λ * param — «сила» регуляризации, которая тянет параметр к 0.

4.5. Регуляризация, переобучение и early stopping

Важно проговорить:

  • Если λ слишком маленький:
    • модель может идеально выучить train, но плохо работать на val;
  • Если λ слишком большой:
    • модель слишком «плоская» и плохо описывает данные (undreft).

Практика:

  • на каждом epoch:
    • считать RMSE на validation;
    • можно делать early stopping:
      • если RMSE_val не улучшается Х эпох подряд — остановиться.

4.6. Как из biased MF сделать топ-K рекомендации

Хотя мы оптимизируем MSE, всё равно можно использовать:

  • для каждого пользователя:
    • считаем \hat{r}_{u,i} для всех (или большого поднабора) items;
    • сортируем по убыванию;
    • берём top-K;
  • считаем Recall@K, NDCG@K по уже знакомому eval-фреймворку.

Обсуждение:

  • MF под MSE оптимизирует «точность чисел», а не ранжирование:
    • две модели могут иметь очень близкий RMSE, но давать заметно разные top-K;
    • иногда модель с чуть большим RMSE даёт лучшую NDCG@K.

Это подводка к тому, зачем нам потом BPR (pairwise) — но сегодня просто фиксируем факт.

5.2. Расширяем до BiasedMF

Шаг 1. Добавляем параметры μ, b_u, b_i:

class BiasedMF:
    def __init__(self, n_users, n_items, n_factors=20, lr=0.01, reg=0.1, n_epochs=10):
        self.n_users = n_users
        self.n_items = n_items
        self.n_factors = n_factors
        self.lr = lr
        self.reg = reg
        self.n_epochs = n_epochs

        self.P = 0.01 * np.random.randn(n_users, n_factors)
        self.Q = 0.01 * np.random.randn(n_items, n_factors)

        # user/item biases
        self.b_u = np.zeros(n_users)
        self.b_i = np.zeros(n_items)

        # global mean
        self.mu = 0.0

Шаг 2. Предсказание:

    def predict(self, u, i):
        return self.mu + self.b_u[u] + self.b_i[i] + np.dot(self.P[u], self.Q[i])

Шаг 3. fit: считаем mu, потом SGD-обучение:

    def fit(self, train_data, val_data=None):
        # train_data: np.array [N, 3]: (u, i, r)

        # считаем глобальное среднее
        self.mu = np.mean(train_data[:, 2])

        for epoch in range(self.n_epochs):
            np.random.shuffle(train_data)
            for u, i, r in train_data:
                u = int(u); i = int(i)
                r_hat = self.predict(u, i)
                e = r - r_hat

                # обновления biases
                self.b_u[u] += self.lr * (e - self.reg * self.b_u[u])
                self.b_i[i] += self.lr * (e - self.reg * self.b_i[i])

                # обновления факторов
                p_u = self.P[u]
                q_i = self.Q[i]

                self.P[u] += self.lr * (e * q_i - self.reg * p_u)
                self.Q[i] += self.lr * (e * p_u - self.reg * q_i)

            if val_data is not None:
                rmse = self.evaluate_rmse(val_data)
                print(f"Epoch {epoch+1}/{self.n_epochs}, val RMSE: {rmse:.4f}")

    def evaluate_rmse(self, data):
        se = 0.0
        for u, i, r in data:
            u = int(u); i = int(i)
            r_hat = self.predict(u, i)
            se += (r - r_hat) ** 2
        return np.sqrt(se / len(data))

На уроке:

  • запускаем обучение на 5–10 эпохах;
  • смотрим, как ведёт себя валидационный RMSE;
  • сравниваем с SimpleMF (без bias):
    • biased MF почти всегда даёт ниже RMSE.

5.3. Простая top-K оценка из biased MF

Для одного пользователя u:

def recommend_top_k(model, user_id, known_items, k=10):
    # known_items: set item’ов, которые пользователь уже оценивал (чтобы при желании их не рекомендовать)
    scores = []
    for i in range(model.n_items):
        if i in known_items:
            continue
        scores.append((i, model.predict(user_id, i)))
    scores.sort(key=lambda x: x[1], reverse=True)
    return [i for i, s in scores[:k]]

Дальше:

  • с помощью eval-фреймворка (из модуля 2) считаем Recall@K или NDCG@K для biased MF,
  • сравниваем с:
    • mean per item,
    • item-based CF,
    • SimpleMF.

8. Резюме урока

В конце стоит проговорить вслух:

  • Biased MF = Simple MF + глобальное среднее + user/item biases.
  • Biases позволяют:
    • модели не тратить латентные факторы на «банальные» смещения;
    • точнее моделировать рейтинги.
  • MF по MSE (explicit MF) хорошо оптимизирует RMSE, но:
    • не всегда даёт идеальное ранжирование;
    • топ-K метрики всё равно нужно измерять отдельно.
  • В следующем уроке мы:
    • переходим от explicit к implicit MF (ALS);
    • учимся работать не с рейтингами, а с логами взаимодействий (view/click/purchase) и confidence weights.
