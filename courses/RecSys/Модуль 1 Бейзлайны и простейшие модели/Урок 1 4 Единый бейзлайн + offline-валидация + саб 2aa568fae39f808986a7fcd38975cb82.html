4. Блок 10–30 мин: теория — candidate generation и scoring

4.1. Архитектура двухэтапного рекоммендера (простая версия)

Объясняешь:

  1. Candidate generation (retrieval)
    • Нам нужно быстро выбрать ограниченное число кандидатов (например, 200–1000 items из всего каталога).
    • Источники кандидатов:
      • популярные товары;
      • item-to-item соседи по последним действиям пользователя;
      • (в будущем: sequence-модели, графы и т.д.).
  1. Scoring & ranking
    • Для каждого кандидата считаем score — сколько он «подходит» пользователю.
    • В простейшем случае:
      • score = w_pop * pop_score + w_cooc * cooc_score + w_recent * recent_score + ...
    • Сортируем по score → берём top-K.

Отдельно проговариваешь:

  • Даже в очень сложных продакшн-системах архитектура часто похожая, просто кандидаты и scoring становятся умнее.

4.2. Как генерим кандидатов в нашем бейзлайне

Мы уже умеем:

  • из популярности:
    • popular_items_global,
    • сегментные списки;
  • из item-to-item:
    • neighbors[item] (список похожих товаров).

Для конкретного пользователя:

  1. Вызываем item-to-item:
    • берём его последние N действий;
    • собираем топ-M1 кандидатов по co-occurrence.
  1. Добавляем популярность:
    • дополняем список до M_total (например, 200) за счёт:
      • сегментных популярных;
      • глобальной популярности.
  1. Убираем дубликаты → получаем candidates.

4.3. Как считать score для кандидатов

Простейшая схема:

  • pop_score(item):
    • нормализованная глобальная популярность (например, log(1 + count));
  • recent_pop_score(item):
    • нормализованный счётчик только по последнему окну;
  • cooc_score(item):
    • суммарный co-occurrence от последних действий пользователя (мы уже считали в 1.3).

Потом:

score(item) = w_pop * pop_score(item) + w_recent * recent_pop_score(item) + w_cooc * cooc_score(item)

Где w_pop, w_recent, w_cooc — ручные гиперпараметры (мы их подберём по ощущениям и по фейковой offline-метрике).

Не обязательно всё нормировать идеально; в первом приближении можно:

  • pop_score = просто count;
  • recent_pop_score = count в недавнем окне;
  • cooc_score = cooc от 1.3.

Главное — схема, не точные значения.

5. Блок 30–55 мин: кодим объединённый рекоммендер

5.1. Вспоминаем, что уже есть

Пусть к этому уроку у тебя уже реализовано:

  • popular_items_global + item_popularity_dict;
  • item_recent_popularity_dict или weighted_popularity_dict;
  • segment_to_items (или хотя бы просто глобальная популярность);
  • neighbors (item → список (other_item, cooc_value));
  • user_history (user → упорядоченный список (item, timestamp)).

5.2. Функция candidate generation

Пишем отдельную функцию:

from collections import defaultdict

def generate_candidates(user_id,
                        n_recent=5,
                        max_from_i2i=200,
                        max_total=300):
    """
    Генерация кандидатов для пользователя:
    - item-to-item на основе последних n_recent действий
    - популярность для дотягивания до max_total
    """
    history = user_history.get(user_id, [])
    seen_items = set(i for i, _ in history)

    candidates_scores_cooc = defaultdict(float)

    # 1. Item-to-item кандидаты
    if history:
        recent_items = [item for item, ts in history[-n_recent:]]

        for idx, item in enumerate(reversed(recent_items)):
            recency_weight = 0.9 ** idx  # как в 1.3, можно менять
            for neighbor_item, c in neighbors[item]:
                candidates_scores_cooc[neighbor_item] += recency_weight * c

    # сортируем item-to-item кандидатов и берем top-N
    i2i_candidates_sorted = sorted(
        candidates_scores_cooc.items(),
        key=lambda x: x[1],
        reverse=True
    )
    i2i_candidates = [item for item, score in i2i_candidates_sorted[:max_from_i2i]]

    # 2. Добавляем популярных, чтобы расширить список
    candidates = list(dict.fromkeys(i2i_candidates))  # уникальные, сохраняем порядок

    for popular_item in popular_items_global:
        if len(candidates) >= max_total:
            break
        if popular_item not in candidates:
            candidates.append(popular_item)

    # убираем уже просмотренные/купленные товары, если нужно по постановке
    candidates = [item for item in candidates if item not in seen_items]

    return candidates, candidates_scores_cooc  # возвращаем и сырые cooc-score'ы

Обсуждаешь:

  • почему полезно возвращать candidates_scores_cooc — для дальнейшего scoring;
  • где можно добавить сегментную популярность (перед глобальной).

5.3. Функция scoring и итоговый recommend

Теперь делаем общий score:

import math

def pop_score(item_id):
    return item_popularity_dict.get(item_id, 0)

def recent_pop_score(item_id):
    return item_recent_popularity_dict.get(item_id, 0)

def score_item(item_id,
               cooc_scores_dict,
               w_pop=1.0,
               w_recent=1.0,
               w_cooc=1.0):
    # cooc_score - что мы вернули в candidates_scores_cooc
    cooc_sc = cooc_scores_dict.get(item_id, 0.0)
    g = pop_score(item_id)
    r = recent_pop_score(item_id)

    return w_pop * g + w_recent * r + w_cooc * cooc_sc

И финальная функция:

def recommend_combined(user_id, k=10,
                       w_pop=1.0, w_recent=1.0, w_cooc=1.0):
    candidates, cooc_scores = generate_candidates(user_id)

    if not candidates:
        # совсем нет истории + мало данных -> fallback на популярность
        return popular_items_global[:k]

    scored = [
        (item, score_item(item, cooc_scores, w_pop, w_recent, w_cooc))
        for item in candidates
    ]

    ranked = sorted(scored, key=lambda x: x[1], reverse=True)
    recs = [item for item, sc in ranked[:k]]
    return recs

6. Блок 55–80 мин: простая offline-валидация

Здесь не надо делать идеальную ML-статью, нужно дать:

  • time-based split,
  • HitRate@K,
  • умение сравнивать два бейзлайна.

6.1. Time-based split

Идея:

  1. Берём всю trainтаблицу с timestamp.
  1. Выбираем split_time:
    • например, 80-й перцентиль timestamp’ов.
  1. Разделяем:
    • train_part = train[timestamp < split_time]
    • val_part = train[timestamp >= split_time]

В коде:

all_events = train.sort_values('timestamp')

split_time = all_events['timestamp'].quantile(0.8)

train_part = all_events[all_events['timestamp'] < split_time]
val_part = all_events[all_events['timestamp'] >= split_time]

Важно объяснить:

  • Модели (популярность, co-occ) нужно считать только на train_part.
  • val_part используется только для проверки качества.

Значит, надо пересчитать:

  • item_popularity_dict,
  • item_recent_popularity_dict,
  • neighbors,
  • user_history

    на основе train_part, а оффлайн-метрику считать по val_part.

(На уроке можно сделать более простую версию и сказать, что «в идеале надо всё пересчитать строго на train_part, домашкой можно дать доработать».)

6.2. Простая HitRate@K

Объяснение студентам:

HitRate@K = доля случаев, когда правильный item оказался в наших top-K рекомендациях.

Упрощённый сценарий:

  • Для каждого пользователя в val_part:
    • берём какой-то целевой event (например, первую покупку после split_time или последний event в val);
    • строим рекомендации по истории до этого события (то есть по train_part);
    • проверяем, попал ли target item в top-K.

Пример (упрощённый):

import numpy as np

K = 10

def build_user_histories(df):
    df = df.sort_values('timestamp')
    uh = defaultdict(list)
    for row in df.itertuples():
        uh[row.user_id].append((row.item_id, row.timestamp))
    return uh

# пересчитываем user_history только на train_part
user_history_train = build_user_histories(train_part)

# выбираем по одному target-событию на пользователя из val_part
val_part_sorted = val_part.sort_values('timestamp')
user_targets = val_part_sorted.groupby('user_id').head(1)  # например, первое событие в val

def hitrate_at_k(recommender_fn, user_history, user_targets, k=10, max_users=1000):
    hits = 0
    total = 0

    for row in user_targets.itertuples():
        user_id = row.user_id
        target_item = row.item_id

        # история до split: берём из user_history_train
        history = user_history.get(user_id, [])
        if not history:
            continue  # нет истории - пропускаем

        recs = recommender_fn(user_id, k=k)

        if target_item in recs:
            hits += 1
        total += 1

        if total >= max_users:  # ограничиваемся для скорости
            break

    return hits / total if total > 0 else None

Теперь можно посчитать:

hr_pop = hitrate_at_k(recommend_popularity, user_history_train, user_targets, k=10)
hr_i2i = hitrate_at_k(recommend_item_item, user_history_train, user_targets, k=10)
hr_comb = hitrate_at_k(recommend_combined, user_history_train, user_targets, k=10)

print("HitRate@10 Popularity:", hr_pop)
print("HitRate@10 Item2Item:", hr_i2i)
print("HitRate@10 Combined:", hr_comb)

И обсудить:

  • часто ли Combined действительно лучший;
  • сколько вообще получились значения (0.01 vs 0.1 vs 0.3 и т.д.);
  • даже если значения маленькие → главное, что можно сравнивать методы между собой.

7. Блок 80–90 мин: сабмит + домашка

7.1. Идея сабмита

Важно:

  • Формат соревнования может быть любой, но принцип один:
    • есть список «сущностей» (user_id / session_id / и т.п.);
    • для каждой сущности нужно выдать список из K item_id в нужном формате.

Условный пример:

test = pd.read_csv("../data/test_interactions.csv")  # или просто список user_id

unique_users_test = test['user_id'].unique()

rows = []
for u in unique_users_test:
    recs = recommend_combined(u, k=10)
    recs_str = " ".join(map(str, recs))  # если нужно в одну строку
    rows.append((u, recs_str))

submit_df = pd.DataFrame(rows, columns=['user_id', 'item_id_list'])
submit_df.to_csv("submission.csv", index=False)

Проговори:

  • Нужно внимательно читать формат:
    • разделитель списка (пробел/запятая);
    • порядок колонок;
    • должны ли быть ровно K item’ов;
    • можно ли повторять item’ы.
