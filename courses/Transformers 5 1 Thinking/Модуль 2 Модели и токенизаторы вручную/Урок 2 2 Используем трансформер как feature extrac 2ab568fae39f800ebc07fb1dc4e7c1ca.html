Урок 2.2. Используем трансформер как feature extractor

Окей, теперь сделаем из 2.2 тоже полноценный урок. Здесь уже немного «взрослый» уровень: будем использовать трансформер как feature extractor и сверху прикручивать свой классификатор.

0. Что мы делаем в этом уроке

Цель урока:

  • понять, как взять предобученный трансформер без головы,
  • вытащить из него эмбеддинги текстов,
  • обучить на этих эмбеддингах свой классификатор (например, LogisticRegression),
  • применить его к новым тикетам поддержки.

Контекст:

В реальной жизни чаще всего у тебя нет готовой модели ровно под твои классы (bug, feature, billing, other), но у тебя есть сильный предобученный трансформер. Логичный шаг — использовать его как «мощный embedder», а сверху ставить свои модели.

1. Зачем вообще нужен feature extractor

Коротко, когда это полезно:

  1. Нужны свои классы, которых нет в готовых моделях с Hub:
    • тикеты техподдержки с внутренней схемой меток;
    • твои тематики новостей;
    • свои категории отзывов.
  1. Данных мало:
    • полноценный fine-tuning большой модели может быть больно дорог,
    • проще «заморозить» трансформер и обучить маленькую голову отдельно.
  1. Хочешь быстро поэкспериментировать:
    • модель даёт устойчивые эмбеддинги;
    • можно протестировать разные простые модели поверх (LogReg, SVM, деревья/ансамбли).

Ментальная модель:

трансформер → превращает текст в вектор признаков (embedding)

наш классификатор → решает задачу на этих признаках.

2. Теория: AutoModel vs AutoModelForSequenceClassification

2.1. AutoModel

AutoModel — это «голый» трансформер без головы задачи.

  • На вход: input_ids, attention_mask и т.д.
  • На выход:
    • last_hidden_state: [batch_size, seq_len, hidden_size]
    • (опционально) hidden_states, attentions, если просим.

last_hidden_state[i, j] — вектор для j-го токена i-го текста.

2.2. Как получить один вектор на текст

Нам нужен один эмбеддинг на текст, а не на каждый токен. Основные варианты:

  1. [CLS]-токен:
    • для BERT-подобных моделей [CLS] (первый токен) служит агрегированным представлением.
    • Берём last_hidden_state[:, 0, :].
  1. Пулинг по токенам:
    • среднее по всем не-паддинговым токенам;
    • максимум по токенам;
    • что-то кастомное.

В этом уроке будем использовать [CLS]-эмбеддинги, как самый простой и стандартный вариант.

2.3. AutoModelForSequenceClassification

AutoModelForSequenceClassification:

  • внутри содержит тот же AutoModel,
  • но сверху добавляет:
    • Dropout + Linearголову → logits,
  • и инициализируется под конкретное num_labels.

В 2.1 мы работали именно с ним.

В 2.2 мы берём AutoModel, сами достаём эмбеддинги и учим внешнюю голову.

3. Практика, часть 1: вытаскиваем CLS-эмбеддинги

3.1. Подготовка

Берём базовую модель, например distilbert-base-uncased:

import torch
from transformers import AutoTokenizer, AutoModel

model_id = "distilbert-base-uncased"

tokenizer = AutoTokenizer.from_pretrained(model_id)
base_model = AutoModel.from_pretrained(model_id)

(Если хочешь русские примеры — можно взять cointegrated/rubert-tiny или любой другой русскоязычный чекпоинт, структура та же.)

3.2. Наши тексты (тикеты)

Сделаем небольшой набор тикетов:

tickets = [
    "The app crashes when I try to upload a photo.",
    "Can you add dark mode, please?",
    "I was charged twice this month.",
    "Great app, just wanted to say thanks!",
]

3.3. Токенизация

enc = tokenizer(
    tickets,
    padding=True,          # выравниваем по длине
    truncation=True,       # обрезаем слишком длинные
    return_tensors="pt",   # PyTorch тензоры
)

print(enc.keys())
print("input_ids shape:", enc["input_ids"].shape)
print("attention_mask shape:", enc["attention_mask"].shape)

Ожидаемо:

  • input_ids: [batch_size, seq_len] → [4, seq_len_max].
  • attention_mask: [4, seq_len_max].

3.4. Прогон через AutoModel

with torch.no_grad():
    outputs = base_model(**enc)

print(outputs.keys())
# dict_keys(['last_hidden_state'] или ещё 'hidden_states', если включали)

Берём last_hidden_state:

last_hidden_state = outputs.last_hidden_state
print("last_hidden_state shape:", last_hidden_state.shape)

Форма:

  • [batch_size, seq_len, hidden_size]

    Например, [4, 20, 768].

3.5. Достаём CLS-эмбеддинги

Предполагаем, что tokenizer добавляет [CLS] первым токеном, и именно его вектор — агрегированное представление:

cls_embeddings = last_hidden_state[:, 0, :]  # [batch_size, hidden_size]
print("cls_embeddings shape:", cls_embeddings.shape)

Это уже матрица признаков:

  • строки — тексты,
  • столбцы — фичи размерности hidden_size (например, 768).

Чтобы использовать в sklearn:

X = cls_embeddings.numpy()
print("X shape for sklearn:", X.shape)

4. Практика, часть 2: учим простой классификатор поверх эмбеддингов

Теперь сделаем маленький кастомный классификатор тикетов.

4.1. Придумываем классы

Допустим:

  • 0 = bug — баги/краши;
  • 1 = feature — запросы фич;
  • 2 = billing — проблема с оплатой;
  • 3 = other — всё остальное (отзывы/благодарности).

Присвоим метки нашим 4 тикетам:

import numpy as np

labels = np.array([
    0,  # "The app crashes..." -> bug
    1,  # "Can you add dark mode..." -> feature
    2,  # "I was charged twice..." -> billing
    3,  # "Great app..." -> other
])

Да, это очень маленький датасет, в реальной жизни их должны быть сотни/тысячи, но для иллюстрации норм.

4.2. Logistic Regression как голова

Поставим sklearn:

pip install scikit-learn

И дальше в коде:

from sklearn.linear_model import LogisticRegression

X = cls_embeddings.numpy()  # [4, hidden_size]
y = labels                  # [4]

clf = LogisticRegression(max_iter=1000)
clf.fit(X, y)

Тут важно объяснить:

  • трансформер заморожен (мы не обучаем его),
  • LogisticRegression учится на уже готовых эмбеддингах,
  • в реальном проекте ты бы делал train/val split и нормальный пайплайн.

5. Практика, часть 3: проверяем на новых тикетах

Создадим новые тикеты, которые модель не видела при обучении головы:

new_tickets = [
    "The app crashes on launch.",
    "Please add the option to change language.",
    "I don't understand this charge on my credit card.",
    "This is the best productivity app I've ever used!",
]

5.1. Вытаскиваем CLS-эмбеддинги для новых текстов

enc_new = tokenizer(
    new_tickets,
    padding=True,
    truncation=True,
    return_tensors="pt",
)

with torch.no_grad():
    outputs_new = base_model(**enc_new)

cls_new = outputs_new.last_hidden_state[:, 0, :]  # [batch_new, hidden_size]
X_new = cls_new.numpy()

5.2. Предсказываем классы

preds = clf.predict(X_new)
probs = clf.predict_proba(X_new)  # вероятности по классам

id2label_custom = {0: "bug", 1: "feature", 2: "billing", 3: "other"}

for text, pred, prob_vec in zip(new_tickets, preds, probs):
    label = id2label_custom[pred]
    top_prob = max(prob_vec)
    print("TEXT:", text)
    print(f"PRED: {label} (conf={top_prob:.3f})")
    print("CLASS PROBS:")
    for class_id, p in enumerate(prob_vec):
        print(f"  {id2label_custom[class_id]}: {p:.3f}")
    print("-" * 60)

Что важно показать ученикам:

  • мы полностью контролируем схему классов;
  • модель трансформера не «знает» про bug/feature/billing — она просто даёт ембеддинги;
  • наша голова на этих эмбеддингах учится отличать категории.

6. Оборачиваем всё в функции (как в реальном проекте)

В реальном коде так писать не будут, конечно, всё будет аккуратно обёрнуто. Покажем минимально адекватный вариант.

6.1. Функция для получения CLS-эмбеддингов

def get_cls_embeddings(texts, tokenizer, model, device="cpu", batch_size=8):
    """
    texts: list[str]
    return: np.ndarray [len(texts), hidden_size]
    """
    import numpy as np

    all_embeddings = []

    model = model.to(device)
    model.eval()

    # простой batching
    for i in range(0, len(texts), batch_size):
        batch_texts = texts[i:i+batch_size]
        enc = tokenizer(
            batch_texts,
            padding=True,
            truncation=True,
            return_tensors="pt",
        ).to(device)

        with torch.no_grad():
            outputs = model(**enc)
        cls_emb = outputs.last_hidden_state[:, 0, :]  # [batch, hidden_size]
        all_embeddings.append(cls_emb.cpu().numpy())

    return np.vstack(all_embeddings)

6.2. Тренируем классификатор в виде отдельной функции

from sklearn.linear_model import LogisticRegression

def train_ticket_classifier(texts, labels, tokenizer, base_model, device="cpu"):
    X = get_cls_embeddings(texts, tokenizer, base_model, device=device)
    y = np.array(labels)

    clf = LogisticRegression(max_iter=1000)
    clf.fit(X, y)
    return clf

6.3. Функция для предсказания

def predict_ticket_classes(texts, tokenizer, base_model, clf, id2label, device="cpu"):
    X = get_cls_embeddings(texts, tokenizer, base_model, device=device)
    preds = clf.predict(X)
    probs = clf.predict_proba(X)

    results = []
    for text, pred, prob_vec in zip(texts, preds, probs):
        label = id2label[pred]
        result = {
            "text": text,
            "label": label,
            "probs": {
                id2label[i]: float(p) for i, p in enumerate(prob_vec)
            }
        }
        results.append(result)
    return results

6.4. Используем всё вместе

# Обучение
train_texts = tickets
train_labels = labels  # 0..3

id2label_custom = {0: "bug", 1: "feature", 2: "billing", 3: "other"}

clf = train_ticket_classifier(
    train_texts,
    train_labels,
    tokenizer,
    base_model,
    device="cpu",  # можно "cuda", если есть
)

# Предсказание
test_texts = new_tickets

results = predict_ticket_classes(
    test_texts,
    tokenizer,
    base_model,
    clf,
    id2label=id2label_custom,
    device="cpu",
)

for r in results:
    print("TEXT:", r["text"])
    print("PRED:", r["label"])
    print("PROBS:", r["probs"])
    print("-" * 50)

Это уже «нормальный» мини-пайплайн, который можно положить в модуль и переиспользовать.

7. Обсуждаем ограничения и подводные камни

Важно честно сказать ученикам, что:

  1. Слишком мало данных → переобучение.
    • Наш пример с 4 тикетами — иллюстрация.
    • В реальности нужен нормальный размер датасета.
  1. Трансформер не дообучается.
    • Мы не обновляем его веса.
    • Если распределение данных сильно отличается, иногда выгоднее делать полноценный fine-tuning (будет дальше по курсу).
  1. Нормализация признаков.
    • Для некоторых моделей (SVM, линейных) может помочь StandardScaler.
    • Можно добавить этап препроцессинга признаков (но не обязательно для начала).
  1. Скорость.
    • Вытаскивать эмбеддинги трансформера — всё ещё дорого.
    • В проде часто:
      • кэшируют эмбеддинги для часто повторяющихся текстов,
      • используют более маленькие модели (distil/mini).
