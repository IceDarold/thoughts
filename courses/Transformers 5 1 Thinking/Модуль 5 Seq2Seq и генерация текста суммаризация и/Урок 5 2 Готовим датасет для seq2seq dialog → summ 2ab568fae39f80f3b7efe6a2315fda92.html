Урок 5.2. Готовим датасет для seq2seq: dialog → summary + DataCollatorForSeq2Seq

Давай, теперь разложим урок 5.2 так же подробно, как 5.1.

Тут мы уже не просто “кормим диалог в модель”, а готовим датасет пар dialog → summary для будущего обучения.

1. Цель урока

После этого урока ученик умеет:

  • понимать, как выглядят данные для seq2seq задач: входной текст и целевой текст;
  • загружать такой датасет в datasets из CSV;
  • писать preprocess_function, которая:
    • токенизирует и вход, и таргет,
    • добавляет labels в batched-стиле;
  • использовать DataCollatorForSeq2Seq, чтобы:
    • аккуратно паддить input’ы и labels,
    • ставить 100 на паддинг в labels.

Фокус только на данных. Обучать будем уже в 5.3.

2. Мини-теория: чем датасет seq2seq отличается от классификации

2.1. Для классификации (что мы делали раньше)

У нас была схема:

  • вход: text (одна строка),
  • выход: один label (класс: bug, feature, billing, other).

После препроцессинга:

  • input_ids, attention_mask, labels (одно число).

2.2. Для seq2seq (суммаризация/перевод)

Здесь задача вида:

  • вход: текст (диалог/документ) → input_text,
  • выход: другой текст (резюме/перевод/ответ) → target_text.

Пример:

input (dialog):
"User: Hi, the app crashes when I upload photos...
 Support: ..."

target (summary):
"User reports crashes when opening gallery after update on iPhone 13, iOS 17.1."

После препроцессинга у каждого объекта должны быть:

  • input_ids, attention_mask — токены входа,
  • labels — токены таргетного текста (в том же словаре токенизатора).

Важно:

В обучении seq2seq используется teacher forcing: на шаге t декодер видит правильные предыдущие токены, а не свои предсказанные, и учится предсказывать следующий.

3. Исходные данные: CSV dialogs.csv

Предположим, у нас есть CSV:

dialog,summary
"User: Hi, I updated the app yesterday and now it crashes...\nSupport: ...","User reports crashes in gallery after update on iPhone 13, iOS 17.1, fix in progress."
"User: I was double charged for my subscription...\nSupport: ...","User was double charged for subscription; support explains refund process."
...

То есть:

  • колонка dialog — длинная строка с диалогом (с '\n' внутри),
  • колонка summary — короткое резюме.

5. Загружаем токенизатор для seq2seq-модели

Как и в 5.1, используем, например, BART (можно T5, суть такая же):

from transformers import AutoTokenizer

model_name = "facebook/bart-large-cnn"
tokenizer = AutoTokenizer.from_pretrained(model_name)

Для T5 есть стиль: "summarize: <текст>", об этом можно отдельно сказать — но для урока достаточно базового.

6. preprocess_function: токенизируем dialog и summary

6.1. Границы длины

Решаем:

  • входы (dialog) ограничиваем, скажем, 512 токенами,
  • таргеты (summary) — 64 токенами.
max_input_length = 512
max_target_length = 64

6.2. Batched-функция для map

Нам нужно:

  1. Взять batch["dialog"] как список текстов,
  1. Взять batch["summary"] как список таргетов,
  1. Закодировать входы токенизатором,
  1. Закодировать таргеты тем же токенизатором,
  1. Поместить labels в результат.

Вариант (современный, рекомендованный HF) — использовать аргумент text_target:

def preprocess_function(batch):
    inputs = batch["dialog"]
    targets = batch["summary"]

    # Токенизация входного текста
    model_inputs = tokenizer(
        inputs,
        max_length=max_input_length,
        truncation=True,
    )

    # Токенизация таргета (summary)
    # Важно: text_target говорит токенизатору, что это таргет.
    labels = tokenizer(
        text_target=targets,
        max_length=max_target_length,
        truncation=True,
    )

    model_inputs["labels"] = labels["input_ids"]
    return model_inputs

Разбор:

  • batch["dialog"] — список строк длиной batch_size.
  • tokenizer(...) с max_length+truncation=True:
    • режет слишком длинные тексты,
    • возвращает словарь с:
      • input_ids — список списков,
      • attention_mask.
  • labels["input_ids"] — уже список списков индексов для таргет-текста.

6.3. Применяем map ко всем сплитам

tokenized_dataset = dataset.map(
    preprocess_function,
    batched=True,
    remove_columns=["dialog", "summary"],
)

print(tokenized_dataset)
print(tokenized_dataset["train"][0])

Теперь у каждого примера есть:

  • input_ids
  • attention_mask
  • labels

Важно:

Мы пока не паддим до одинаковой длины — это сделает DataCollatorForSeq2Seq.

7. Что реально лежит внутри tokenized_dataset

Посмотрим:

example = tokenized_dataset["train"][0]

print(example.keys())
# dict_keys(['input_ids', 'attention_mask', 'labels'])

print(len(example["input_ids"]), len(example["labels"]))
print(example["input_ids"][:10])
print(example["labels"][:10])

Что объяснить:

  • input_ids и labels — это списки (Python list), пока не тензоры.
  • input_ids могут быть длиной до max_input_length,
  • labels — до max_target_length.

Это норм: датасет обычно хранит списки, а тензоры мы получим уже на этапе формирования батча.

8. DataCollatorForSeq2Seq: красивый батчинг и паддинг

8.1. Зачем нужен DataCollatorForSeq2Seq

Когда Trainer/DataLoader собирает батчи, он получает список примеров:

[
  {'input_ids': [...], 'attention_mask': [...], 'labels': [...]},
  {'input_ids': [...], 'attention_mask': [...], 'labels': [...]},
  ...
]

Нужно:

  • подогнать длину всех input_ids в батче до одной (паддинг),
  • то же для labels,
  • превратить всё это в torch.Tensor.

Для seq2seq ещё важно:

  • у labels на позициях паддинга должен быть 100 (чтобы лосс их игнорировал).

DataCollatorForSeq2Seq всё это делает за нас.

8.2. Создаём collator

from transformers import DataCollatorForSeq2Seq, AutoModelForSeq2SeqLM

model = AutoModelForSeq2SeqLM.from_pretrained(model_name)

data_collator = DataCollatorForSeq2Seq(
    tokenizer=tokenizer,
    model=model,
    label_pad_token_id=-100,  # по умолчанию -100
)

Разбор:

  • tokenizer нужен, чтобы знать pad token и т.п.
  • model иногда используется collator’ом для спец.-штук (например, decoder_start_token_id).
  • label_pad_token_id=-100:
    • на позициях паддинга в labels collator будет ставить 100,
    • потом в loss эти позиции игнорируются.

9. Проверяем работу collator’а на практике

Чтобы показать ученикам, что именно делает collator, можно собрать батч руками через DataLoader.

from torch.utils.data import DataLoader

train_loader = DataLoader(
    tokenized_dataset["train"],
    batch_size=4,
    shuffle=True,
    collate_fn=data_collator,
)

batch = next(iter(train_loader))

for k, v in batch.items():
    print(k, v.shape)

Ожидаемый вывод:

input_ids      torch.Size([4, max_input_len_in_this_batch])
attention_mask torch.Size([4, max_input_len_in_this_batch])
labels         torch.Size([4, max_target_len_in_this_batch])

Можно заглянуть внутрь:

print(batch["input_ids"][0])
print(batch["labels"][0])

И важно объяснить:

  • там, где паддинг во входах (input_ids) → токен паддинга;
  • там, где паддинг в labels → 100.

Для наглядности:

pad_token_id = tokenizer.pad_token_id
print("pad_token_id:", pad_token_id)

print("First labels tensor:", batch["labels"][0])
print("Labels != -100:", (batch["labels"][0] != -100).sum().item(), "tokens used in loss")

10. Типичные ошибки и грабли

Полезно сразу проговорить, чтобы ребята не ловили те же баги по сто раз:

  1. Забыли remove_columns в map
    • Если оставить dialog и summary вместе с input_ids/labels,

      Trainer может ругаться, что в батче есть неожиданные поля.

    • Лучше удалять оригинальные текстовые колонки, а сырые тексты хранить отдельно, если нужно.
  1. Попутали input_text и target_text
    • Некоторые новички делают labels = batch["label"], как в классификации.

      Для seq2seq labels — это токены таргетного текста, а не класс.

  1. Слишком агрессивный тримминг
    • max_input_length=128 для длинных диалогов → summary может терять контекст (начало/конец).
    • Иногда имеет смысл:
      • или поднять max_input_length,
      • или заранее обрезать диалог до “важной части”.
  1. Модель и токенизатор не совпадают
    • Надо использовать один и тот же model_name для AutoTokenizer и AutoModelForSeq2SeqLM.
    • Иначе словарь не совпадёт с embedding-матрицей.
