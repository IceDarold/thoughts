М0. Онбординг грандмастер-табличника и 5-часовой спринт

1) Цели модуля

  • Сформировать личный спринт-протокол на 5 часов (поминутный план, чек-листы).
  • Развернуть production-скелет проекта (шаблон репо, CLI-скрипты, конфиги).
  • Освоить минимально достаточную валидацию без утечек и быстрый бейзлайн ≤30 минут.
  • Научиться контролировать CV↔LB и быстро чинить расхождения.
  • Сделать один полный мини-спринт по реальному Kaggle кейсу.

2) Формат и тайминг

  • 2 занятия × 2 часа (теория + лайв-код разбор)
  • 1 самостоятельный 5-часовой спринт (в ближайшие 2–3 дня после занятий)
  • Итог: защита протокола (10 мин) + короткий разбор.

3) Предварительные требования (проверка на входе, 10–15 мин)

  • Установлены: Python ≥3.10, Git, Kaggle API, Poetry/uv/Conda (любой менеджер), gcc/clang.
  • Библиотеки: polars/pyarrow, numpy, pandas, scikit-learn, lightgbm, xgboost, catboost, optuna, joblib, matplotlib.
  • Аккаунт Kaggle, доступ к соревнованиям.
  • Короткий pre-quiz: чем StratifiedKFold отличается от GroupKFold? Когда нужен TimeSeriesSplit?

Принципы

  • Полная воспроизводимость ≤15 минут на чистой машине: make_baseline.sh + config.yaml.
  • OOF-first: сохраняем OOF-предсказания и метрики для контроля CV↔LB.
  • Быстрые I/O: Parquet + polars (ленивые сканы), кеширование промежуточных фич.
  • Единый CLI (argparse/typer): одинаковые флаги для моделей и сплитов.
  • Логгинг времени: декоратор @timed; суммарный отчёт по шагам.

5) Спринт-протокол (минуты и чек-листы)

Первые 60 минут (золотой час)

  • 0–10: прочитать условие, метрику, лимиты сабмитов; понять цель (классификация/регрессия/ранжирование/TS).
  • 10–20: быстрый просмотр данных (schema, nrows, nan-карта, типы, группирующие ключи, явные временные/групповые поля).
  • 20–30: поднять бейзлайн (CatBoost или LightGBM) на разумной CV-схеме, получить OOF и 1-й сабмит.
  • 30–45: проверить CV↔LB корреляцию (хотя бы по валидационной доле) + adversarial validation (быстрая версия).
  • 45–60: добавить 3–5 безопасных фич (частоты, простые агрегаты, базовые лаги/роллинги при TS), 2-й сабмит.

Остальные 4 часа

  • 60–120: системный feature factory (groupby-агрегаты, взаимодействия, таргет-кодировки со сглаживанием по фолдам).
  • 120–180: стабилизация CV (альтернативный сплит: Group/Time/Purged; настройка параметров, поиск порога/калибровка).
  • 180–240: лёгкий ансамбль (2–3 модели, веса по OOF; bagging seeds), финальный сабмит + «паник-кнопка».
  • Паник-кнопка (если LB просел): откатить рискованные фичи, сменить сплит, уменьшить learning_rate+увеличить деревья, проверить dtype/категории, выкинуть «подозрительные» фичи через adversarial importance.

Чек-лист утечек (краткий)

  • Нет ли признаков, агрегированных с использованием будущего?
  • Кодировки по таргету — только внутри фолда.
  • В TS: лаги и статистики не заглядывают вперёд, валидируем по времени.
  • Группы не «рвём» между train/valid.
  • Нельзя использовать идентификаторы, сливающие таргет (hash-ключи, порядковые ID без внимания).

6) Минимально достаточная валидация

  • Выбор из: StratifiedKFold / GroupKFold / TimeSeriesSplit / PurgedKFold — по структуре данных.
  • Быстрый adversarial validation (модель «train vs test»): если AUC > 0.6 — готовьтесь к дрифту; корректируйте сплит или фичи.
  • Контроль стабильности: std OOF по фолдам, сплит-перестановки, корреляция OOF↔LB (хоть и грубо на сабмитах).

7) Быстрые бейзлайны (пресеты)

  • Классификация: CatBoost (auto cat handling), LGBM (hist-бустинг), калибровка порога для F1/AUC-ориентиров.
  • Регрессия: LightGBM с монотонимическими ограничениями (если есть смысл), лог-трансформ таргета.
  • Time series как табличка: групповые лаги/роллинги, GroupKFold по сущностям + holdout по последним периодам.
  • Ранжирование/CTR-подобные: LGBM ranker/регрессия с последующим top-k отбором и пост-нормировкой.

8) Практикум занятия №1 (90–120 мин)

Кейс: Home Credit (кредитный риск) или Porto Seguro (безопасный водитель).

Цель: с нуля до осмысленного сабмита за ≤60 минут, затем добавить 3–5 безопасных фич.

Шаги:

  1. Инициализируем шаблон, настраиваем config.yaml, проверяем Kaggle API.
  1. Запускаем make_baseline.sh → получаем OOF и сабмит_1.
  1. Делаем короткий adversarial validation.
  1. Добавляем частоты/сглаженные target-кодировки по fold’ам → сабмит_2.
  1. Сравнение OOF vs LB, фиксируем в паспорт решения (1 страница).

9) Практикум занятия №2 (90–120 мин)

Кейс: IEEE-CIS Fraud Detection или Santander Customer Transaction (анонимные фичи).

Цель: показать, как не скатиться в «магические признаки» и как держать CV стабильным.

Шаги:

  1. Два альтернативных сплита (StratifiedKFold vs Group/Time-aware) → сравнить std OOF.
  1. «Рискованный приём» (например, row-wise статистики/кластерные признаки) → проверить перенос на hold-out.
  1. Лёгкое блендирование LGBM+CatBoost (веса по OOF) → сабмит_финал.
  1. Разбор «паник-кнопки» на примере намеренно «сломленного» сабмита.

10) Самостоятельный 5-часовой спринт

Выбрать одно (учитель назначает):

  • Классификация: Home Credit / Porto Seguro / IEEE-CIS Fraud.
  • Анонимные признаки: Santander Customer Transaction.
  • Регрессия: House Prices (ускоренная версия) или Sberbank Housing (мини-подмножество).

Правила:

  • Макс 3 сабмита.
  • Обязательно: OOF-лог, ноут/скрипты, submissions/ с timestamp, паспорт решения (1 стр).
  • Метрика успеха: улучшение от бейзлайна ≥ X% + CV-стабильность (std OOF ≤ Y), CV↔LB корреляция положительная.

11) Оценивание

  • Содержание (40%): корректная CV, отсутствие явных утечек, чистая структура проекта.
  • Результат (30%): качество относительно бейзлайна, адекватность сабмит-стратегии.
  • Инженерия (20%): воспроизводимость ≤15 мин, разумные времена шагов, аккуратные артефакты (OOF/модели/логи).
  • Рефлексия (10%): паспорт решения, чёткие next steps.

12) Материалы модуля (выдаются)

  • Шаблон репозитория (как выше) + make_baseline.sh.
  • Preset-конфиги для CatBoost/LGBM/XGBoost (классификация/регрессия/ранк).
  • Чек-листы: первые 60 минут, утечки, CV↔LB-диагностика, паник-кнопка.
  • «Памятка Kaggle CLI»: команды для загрузки данных, сабмитов, версионирования.
  • Шпаргалка по типовым CV-схемам (когда какую выбрать) и быстрым фичам (частоты, агрегаты, лаги).

13) Домашнее задание (до спринта)

  1. Развернуть шаблон, положить один публичный датасет, прогнать make_baseline.sh.
  1. Написать свою версию спринт-протокола (на 5 часов) под выбранную метрику.
  1. Сформировать список из 10 безопасных фич по датасету (с комментариями «почему не утечка»).

14) Расширения по желанию (если время останется)

  • Optuna для подбора 5–7 ключевых гиперпараметров (time-boxed 15–20 мин).
  • Монолитный blend.py с калибровкой (Platt/Isotonic) и отбраковкой «вредных» моделей по OOF.
  • Профилировка узких мест: чтение/запись, feature factory, тренировки.

Выход модуля: у каждого ученика готовый репозиторий, персональный спринт-протокол, отработанный 5-часовой мини-спринт и набор чек-листов/пресетов. Это станет фундаментом для всех следующих модулей и финальных контрольных.
