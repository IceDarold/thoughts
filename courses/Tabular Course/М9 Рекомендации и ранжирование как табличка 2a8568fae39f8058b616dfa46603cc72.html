М9. Рекомендации и ранжирование как табличка

ниже — подробный Модуль 9. Фокус: рекомендации и ранжирование “как табличка” — быстрый candidate gen → табличный re-rank (GBDT), честные групповые сплиты и метрики @K. Всё завязано под 5-часовой формат.

1) Цели модуля

  • За ≤10 минут определить структуру запроса (user/session/context) и корректный group-aware сплит.
  • Построить конвейер кандидатов (co-visitation / pop-decay / item2vec / MF/ALS/BPR) и табличный re-rank (LGBM/CB/XGB ranker).
  • Считать метрики @K (Recall/MAP/NDCG, HitRate) и уметь их стабилизировать.
  • Закрыть cold-start (новые пользователи/товары) и бизнес-правила (диверсификация, свежесть, повторные покупки).
  • Собрать лёгкий ансамбль по источникам кандидатов и моделям.

Выход: папка recsys/ с candidate_gen.py, rec_feature_factory.py, rank_metrics.py, rankers/*.py|*.yaml, чек-листы сплитов/утечек.

2) Формат и тайминг

  • 2 занятия × 2 часа (теория + лайв-код).
  • Самостоятельный мини-спринт 4–5 часов.
  • Защита «паспорт решения» (1 стр) + 2 сабмита.

3) Карта решений за ≤10 минут

  1. Что такое “запрос”?
    • user-level (история пользователя), session-level (последние N событий), context-level (поисковая фраза/дестинация/даты).
  1. Что предсказываем?
    • next-item / repeat-purchase / click-through / booking-choice.
  1. Метрика и K: Recall@20/MAP@20/NDCG@20 (или бизнес-метрика).
  1. CV: GroupKFold/GroupShuffleSplit по query_id (user_id или session_id), time-aware (train раньше valid); gap ≥ max_окна.
  1. План: 3–4 источника кандидатов → 200–500 кандидатов/запрос → табличный re-rank → метрики @K → лёгкий ансамбль.

4) Честная валидация (без самообмана)

  • Группы = запросы: не делим один и тот же user/session между train/valid.
  • Time-aware: все статистики и модели кандидатов обучаются только на прошлых событиях относительно валидации.
  • Никаких “future co-vis”: co-visitation/частоты считаем скользящим окном до момента запроса.
  • Нормы @K: метрики считаем на полном списке с учётом “истинных” таргетов в валидном окне.

5) Candidate Generation (минимальный набор + расширения)

База (быстро и крепко):

  • Pop-decay: глобальная/по категории/по региону — частота с экспон. затуханием по времени.
  • Co-visitation (item×item): окна co-occurrence (напр., в одной корзине/сессии), взвешивания по позиции/времени.
  • Recency-aware user-history: последние N просмотренных/купленных → соседи по co-vis/embeddings.
  • Item2vec (skip-gram на сессиях): k ближайших соседей по косинусу.

Средний уровень:

  • MF/ALS/BPR (user×item матрица, implicit feedback): топ-N по score для каждого запроса.
  • Category/brand aware: популярность внутри (cat, brand), совместимость “user profile × item”.
  • Context-aware: фильтрация по дате/цене/району/длительности (Expedia-стайл).

Рекомендации по количеству: суммарно 200–500 кандидатов на запрос (merge-union + de-dup), с тегом источника и score каждого источника.

6) Feature Factory для re-rank (user/item/query/pair)

Глобальные (item):

  • популярность (глоб/категория), свежесть (age), цена/скидка, рейтинг/отзывы, редкость.

Пользовательские (user/profile):

  • частоты/доли категорий, средняя цена, последние покупки, recency/periodicity, region/home-cat, device.

Контекст (query):

  • дата/сезон, длительность/бюджет/локация, search filters, day-of-week, mobile/desktop.

Парные (user×item / session×item):

  • co-vis score, item2vec cos, Jaccard/TF-IDF по категориям/тэгам, цена vs личный медианный чек, distance/overlap предпочтений, “similarity to last-k”.

Source-aware:

  • бинарники “кандидат из X”, исходный score (нормирован/ранг), аггрегаты по источникам (max/min/mean).

Ранж-специфика:

  • позиция в листинге (если есть), penalty за “слишком частые повторки” (anti-spam), diversity-bucket id.

7) Потоки таргетов и негативов

  • Labeling: 1, если item ∈ целевой набор (например, куплен/кликнут в валидном окне запроса), иначе 0.
  • Negatives:
    • полунегативы = кандидаты, не попавшие в “истину”;
    • добавить сложные негативы (pop-top, похожие соседи) — стабилизирует обучение.
  • Баланс: стратифицированный негативный sampling на запрос, фиксировать top-K eval независимо от количества негативов.

8) Модели и лоссы (ранжирование)

  • LightGBM ranker: LambdaMART (objective=rank_xendcg/…); группа = query_id; метрика — NDCG@K/Map@K.
  • CatBoost ranker: YetiRank/PairLogit, group_id=query_id, хорошо ест категориальные.
  • XGBoost rank: pairwise/ndcg; но следим за временем/стабильностью.
  • Pointwise CTR-re-rank (логистическая регрессия/GBDT с BCE) — baseline;
  • Pairwise/listwise предпочтительнее для @K, но строже к корректности групп.

Трюки:

  • нормировать фичи/скоры источников внутри запроса (z/rank);
  • group-wise регуляризация (min_data_in_leaf per group ↑);
  • seed-bagging (3–5 сидов) с усреднением нормированных скоров.

9) Пост-процессинг и бизнес-правила

  • Дедуп и tie-breakers (по source-score/свежести/цене).
  • Diversity: MMR/greedy по категориям/брендам/продавцам; ограничить повтор из одной категории в топ-N.
  • Novelty/serendipity: бонус за новые для пользователя товары (bounded).
  • Repeat-purchase: для FMCG — whitelist для частых повторов (Instacart-стайл).
  • Фильтры наличия/логистики: stock, регион, сроки — как hard constraints.

10) Метрики @K и мониторинг

  • Recall@K / HitRate@K — базовый охват истинных.
  • MAP@K / NDCG@K — положение “истины” в списке (бонус за верх).
  • Coverage/Novelty/Diversity — доля уникальных товаров/категорий.
  • Latency budget — измеряем среднее и P95 у pipeline (cand gen + re-rank).
  • CV↔LB — вести таблицу сабмитов: split, cand-sources, feature set, ranker, @K метрики.

11) Cold-start и дрейф

  • Новый пользователь: pop-decay в нужном bucket (регион/категория/время), context-filters.
  • Новый товар: контентные признаки (категория/бренд/цена/описание-эмбеддинги) + кат/рег-популярность.
  • Дрейф: AV по “старые запросы vs новые”, PSI по ключевым фичам; ускоренное переобучение cand-матриц (ежедневно/еженедельно).

12) Бейзлайны и ансамбль

  • Baseline-0 (≤30–45 мин): pop-decay + co-vis → re-rank pointwise (BCE) → Recall@20/MAP@20.
  • Baseline-1 (≤90 мин): + item2vec кандидаты, + ranker (LambdaMART) → NDCG@20.
  • Ансамбль (30–45 мин):
    • стэкинг: логистическая регрессия по z-скорам нескольких ранкеров;
    • RRF (reciprocal rank fusion) поверх отсортированных списков;
    • веса по OOF-NDCG.

13) Мини-практикум A (60–90 мин): Instacart-стайл repeat

Цель: повторные покупки и @K.

План:

  1. cand: pop-decay per user×department + co-vis корзин.
  1. features: user-freq/recency, item-reorder-rate, pair co-vis score.
  1. ranker: LGBM LambdaMART; Recall@20/MAP@20; короткий анализ ошибок (какие dep проваливаются).

14) Мини-практикум B (60–90 мин): Expedia-стайл выбор варианта

Цель: контекстный фильтр + re-rank.

План:

  1. cand: pop по destination×season + item2vec.
  1. features: price vs user-median, position bias, stay_length, “distance to query”.
  1. CatBoost YetiRank; NDCG@20; post-процедуры: diversity по chain/region.

15) Мини-практикум C (45–60 мин): Avito-стайл спрос/CTR

Цель: pointwise vs pairwise.

План:

  1. cand: co-vis объявления + pop категорий.
  1. features: текст/кат-сигналы (tf-idf-суммы/эмбеддинги как числовые), user-history.
  1. BCE-re-rank vs LambdaMART; сравнить MAP/NDCG.

16) Самостоятельный 5-часовой спринт

Варианты: repeat-покупки / выбор предложения / CTR-реранж.

Требования:

  • Корректный group+time сплит, ≥3 источника кандидатов, 10–20 core-фич + 5 pair-фич.
  • 2 сабмита: (а) pointwise baseline, (б) listwise/pairwise + post-processing (diversity/novelty).
  • Паспорт решения: схема CV, источники кандидатов, ключевые фичи, @K метрики, узкие места по времени.

17) Чек-лист утечек (версия М9)

  • Cand-статистики (co-vis/pop/item2vec/MF) считаются только на train-прошлом; ни одного события из валид/тест.
  • В парных фичах нельзя использовать сигналы “после” момента запроса.
  • Группы не рвутся: один query_id строго в одном сплите.
  • Post-filters/stock — не должны смотреть в будущее.

18) Оценивание

  • Корректность (35%): сплит, отсутствие утечек, честные кандидаты.
  • Результат (30%): Recall/MAP/NDCG@K, стабильность на альтернативном сплите.
  • Инженерия (20%): скорость cand gen и re-rank, кэш Parquet/Polars, reproducibility ≤15 мин.
  • Отчёт (15%): ясная аргументация источников/фич/моделей, план борьбы с дрейфом.

19) Что уходит «в рюкзак»

  • candidate_gen.py: pop-decay, co-vis, item2vec, MF/ALS/BPR, merge-union + нормировка скоров.
  • rec_feature_factory.py: user/item/query/pair фичи, нормировки внутри запроса, anti-spam/diversity hooks.
  • rankers/: пресеты lgbm_rank.yaml, catboost_yeti.yaml, xgb_rank.yaml, скрипт seed-bagging.
  • rank_metrics.py: Recall/MAP/NDCG@K, Coverage/Novelty/Diversity, latency-лог.
  • splits_rec.py: group+time сплиты с gap по истории.
  • Чек-листы: «cand без утечки», «@K sanity-checks», «diversity/novelty правила».

20) Встроено в 5-часовой формат (ритм)

  • 0–30 мин: CV (group+time), cand: pop+co-vis → сбор 300–500 кандидатов, pointwise бейзлайн → сабмит_1.
  • 30–90 мин: + item2vec/MF кандидаты, ядро фич → LambdaMART → сабмит_2.
  • 90–150 мин: пост-процедуры (diversity/novelty), альтернативный сплит, стресс-тест.
  • 150–210 мин: ансамбль (rankers/источники), финальный топ-K.
  • 210–240 мин: паспорт решения: что дало прирост, где риски, что бы делали с ещё 2 часами.

Хочешь — накину каркасы candidate_gen.py и rec_feature_factory.py ровно под твой текущий темплейт (та же CLI-схема, что у CV/Encoder/TS-factory), чтобы команда могла сразу вставить модуль в спринты.
