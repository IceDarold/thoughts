М2. Категориальные признаки без боли

1) Цели модуля

  • Выбирать кодировку под структуру данных за ≤10 минут (decision tree выбора).
  • Собрать «EncoderFactory» с безопасной OOF-логикой (никаких утечек).
  • Научиться приручать high-cardinality: редкие категории, коллизии, unseen.
  • Стабилизировать качество через сглаживания и контроль дрифта.
  • Сравнить минимум 5 подходов на реальных задачах и зацементировать «боевые пресеты».

Выход: готовые скрипты кодировок (one-hot / freq / target/LOO/ME / CatBoost-ordered / hashing / binary / GLMM), чек-лист утечек, decision-matrix выбора.

2) Формат и тайминг

  • 2 занятия × 2 часа (теория + лайв-код).
  • 1 самостоятельный категориальный мини-спринт (3 часа).
  • Короткая защита: «что и почему сработало / не сработало».

3) Карта решений: как выбрать кодировку (≤10 мин)

  1. Кардинальность и вид:
    • cardinality ≤ 15 → one-hot (либо ordinal + монотонные модели).
    • 15 < cardinality ≤ 1000 → freq/count как база; + target/LOO/ME при стабильной CV.
    • > 1000 → hashing (n_buckets 2¹²–2¹⁴), binary, CatBoost-ordered.
  1. Структура задачи:
    • Время/сессии → только time-aware кодировки (roll/expanding counts), purged/embargo.
    • Группы (user/session/item) → GroupKFold; кодировки внутри фолда и/или по времени.
    • Сильный дрифт → AV-проверка; меньше target-зависимых фич, больше freq/hash.
  1. Модель:
    • CatBoost → отдаём «сырые» категориальные или ordinal + cat_features.
    • LGBM/XGB → числовые кодировки (freq/target/hash/binary) + аккуратная регуляризация.

4) База без утечек (must-have)

  • Rare pooling: склеить редкие категории в __RARE__ по порогу (например, кумулятивно до 99% покрытия).
  • One-hot: только для low-card; drop-first; контролировать раздувание матрицы.
  • Freq/Count encoding: относительные частоты по train-fold (OOF!), отдельная таблица для test (left join).
  • Ordinal mapping: фиксированная карта cat→int (train-based), unseen→-1/__UNK__.
  • Типы: хранить как category/int32, беречь память, проверять консистентность категорий между train/test.

5) Продвинутые кодировки (и когда их брать)

  • Target mean / Fold-target (KFold): среднее по таргету внутри фолда.
    • Риск: утечка, если не OOF.
    • Стабилизация: M-estimate / James–Stein сглаживание (λ по частоте группы).
  • LOO (leave-one-out): устойчивее к редким, но шумно на маленьких n; добавляйте шум (Gaussian) + сглаживание.
  • CatBoost-ordered TS: упорядоченная статистика без утечки через «будущее» (особенно для временных).
    • В LGBM имитировать через expanding mean по времени (per-entity).
  • GLMM encoding: регуляризация на уровне категорий через смешанные модели (добавка к линейной части).
    • Времязатратно → как «тяжёлая артиллерия» для стабилизации.
  • Hashing trick: быстрый и устойчивый; настраиваем n_buckets (компромисс: коллизии vs размер).
    • Для текстовых категорий (title, brand) — часто лучший старт.
  • Binary encoding: компромисс между one-hot и hashing.
  • WOE/IV (скоринг): когда нужна интерпретируемость и монотонность; внимание к бинингам и балансам классов.
  • Entity embeddings (NN): мощно при очень high-card и больших данных; в 5-часовом формате — опциональный бонус.

6) High-cardinality: рецепты

  • Top-K + RARE: держим top-K по частоте (K~200–2000 по размеру), остальное в __RARE__.
  • Комбинации категорий: кроссы A×B только через freq/count/hash; избегать one-hot на кроссах.
  • Unseen: явный __UNK__/bucket; для hash — «всё покрывается».
  • Стабильность: M-estimate, clip для таргет-статов (например, в [p05, p95] по фолду).

7) Time-aware кодировки

  • Rolling/expanding counts per entity (user/item/merchant) с embargo/gap.
  • Leak-чек: ни один наблюдаемый таргет будущего периода не попадёт в текущее кодирование.
  • PurgedKFold: при сильной авто-корреляции.
  • Decay-агрегаты: экспоненциальный спад для «свежести» события.

8) Контроль качества и стабильности

  • OOF-первым делом: кодировки считаются только на train-fold; OOF-файл сохраняется.
  • Ablation-тесты: +X фича → Δметрики на OOF, доверительный интервал по folds.
  • Null-importances и permutation: отличаем «шум» от реальной пользы.
  • AV-дрифт: кодировка усилила различия train/test? Сместите в сторону freq/hash.

9) Мини-практикум A (60–90 мин): «Cat in the Dat-стайл»

Цель: сравнить 5 кодировок на одной задаче.

План:

  1. Low-card фичи → one-hot vs ordinal.
  1. Mid-card → freq vs target(KFold) с M-estimate.
  1. High-card → hashing (2¹³) vs CatBoost-ordered.
  1. Таблица: качество OOF, std по фолдам, скорость, риск утечки.

Артефакты: encoder_report.md, OOF-метрики, диаграмма «прибавка/стоимость».

10) Мини-практикум B (60–90 мин): «Анонимки/фрод»

Цель: выжать сигнал из «безымянных» колонок, не уехав в утечку.

План:

  1. Быстрый бейзлайн LGBM с freq.
  1. Добавить LOO + шум + сглаживание; сравнить OOF/LB.
  1. Проверить AV: не вырос ли дрифт?
  1. Дешёвый ансамбль: LGBM(fold-target) + CatBoost(ordered).

11) Мини-практикум C (30–45 мин): time-aware

Цель: энкодинги в продажах/логах.

План: rolling counts per user/item (window=7/28), expanding mean per item, embargo=1 шаг; сравнить с «наивным» target-mean (показать утечку).

12) Самостоятельный мини-спринт (3 часа)

  • Датасет с 2–3 high-card признаками.
  • Требования: ≥3 кодировки, OOF-лог, AV-отчёт, 2 сабмита; короткий отчёт «что оставить в финале и почему».

13) EncoderFactory: скелет (что должно быть в репо)

  • encoders/
    • one_hot.py (low-card), freq.py (универсальный),
    • target_kfold.py (с M-estimate, шум, clip), loo.py,
    • catboost_ordered.py (обёртка/имитация ordered-stat),
    • hashing.py (настраиваемый buckets), binary.py,
    • time_aware.py (rolling/expanding + embargo),
    • glmm.py (опционально).
  • Общий интерфейс: fit(X_train, y_train, groups, fold_idx) → transform(X); сохранение карты категорий, параметров сглаживания, seed.
  • Тесты:
    • отсутствие пересечений (train-only fit),
    • консистентность mapping на test,
    • speed-снэпшот (каждый encoder ≤ N сек на 1e6 строк условно).

14) Чек-лист утечки (версия для М2)

  • Любая статистика по таргету — только OOF (или time-aware expanding).
  • Никаких кросс-фолдов для кодировок (fit на fold-train → transform на fold-valid).
  • Для времени — gap/embargo обязателен.
  • Проверить AV до/после кодировок: не стал ли is_test предсказуемее?

15) Оценивание

  • Корректность (35%): ни одной утечки, верные сплиты/time-политики.
  • Результат (30%): стабильная прибавка к бейзлайну (ΔOOF, ΔLB), низкий std по фолдам.
  • Инженерия (20%): EncoderFactory, сохранённые артефакты (карты, OOF), скорость.
  • Отчёт (15%): ясное «почему этот метод» + план на дрифт.

16) Встроено в 5-часовой спринт

  • 0–20 мин: decision-matrix → выбрать 2 кодировки (safe + «смелая»).
  • 20–60 мин: реализовать их в OOF-пайплайне, получить сабмит_1.
  • 60–120 мин: стабилизация (сглаживания, rare pooling, hash buckets), сабмит_2.
  • 120–240 мин: добавить time-aware/комбинации, быстрый ensemble, финал.

17) Что уходит «в рюкзак»

  • Decision-matrix выбора кодировок.
  • EncoderFactory с готовыми пресетами (freq, target/LOO/ME, hashing, ordered).
  • Шпаргалка по сглаживаниям, rare-порогам, hash-размерам и time-политикам.
  • Шаблон A/B-абляций на OOF с доверительными интервалами.

Если захочешь, соберу минимальный боевой шаблон EncoderFactory (файлы/интерфейсы/тесты) прямо под твой репозиторий, чтобы в М3 сразу использовать без раскачки.
