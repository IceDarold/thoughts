М3. Кредитный скоринг и дисбаланс

1) Цели модуля

  • Уметь распознавать и обрабатывать дисбаланс: class weights, down/up-sampling, focal-loss, threshold moving.
  • Настраивать валидирование для клиентских/временных данных (GroupKFold по клиенту, Time/Purged по транзакциям).
  • Работать с бизнес-метриками: ROC-AUC/PR-AUC, top-K capture, кастомная AMEX-метрика (ранжирующая).
  • Построить «безопасную фабрику фич» для счетов/транзакций: агрегаты per-client/device/merchant с окнами, decay.
  • Собрать лёгкий ансамбль (CatBoost/LGBM/XGB) и корректно калибровать вероятности (Platt/Isotonic/Brier).

Выход: пайплайн «скоринг под дисбаланс» (скрипты веса/порог/калибровка), чек-лист утечек, готовые пресеты.

2) Формат и тайминг

  • 2 занятия × 2 часа (теория + лайв-код).
  • Самостоятельный мини-спринт 5 часов (в окно между занятиями или сразу после).
  • Короткая защита «паспорт решения».

3) Карта решений: что делать при дисбалансе (≤10 минут)

  1. Тип задачи
    • Скоринг/фрод = ранжирование (порог вторичен) → ROC-AUC/PR-AUC, top-K.
    • Операционные правила = нужен порог → оптимизируем по F1/Recall@K/бизнес-функции издержек.
  1. Быстрый выбор техники
    • Мало данных/скорость важна → class_weight / scale_pos_weight (Cat/LGBM/XGB).
    • Сильный дисбаланс и шумные миноры → focal loss (или alpha-weighted BCE).
    • Огромный класс «0» → down-sampling негатива (с «поправкой базовой вероятности») + веса.
    • PR-AUC важнее → контролируем вероятность, калибруем.
  1. CV-схема
    • GroupKFold по client_id (скоринг, мульти-стейтменты).
    • Time/Purged по транзакциям (фрод, лаги/роллинги, embargo).
    • Для кастомных ранжирующих метрик: считать на OOF и на hold-out.

4) Метрики: что и зачем

  • ROC-AUC — устойчивый ориентир при дисбалансе, но может «лгать» про top-K.
  • PR-AUC — ценен при очень редком «1», чувствителен к калибровке.
  • Top-K / Capture@K — «какую долю мошенничества поймаем в верхних K%».
  • Кастом AMEX — ранжирование + capture в топ-х; оптимизируем ранг, не порог.
  • Brier / ECE — для калибровки (если нужна пороговая политика).

5) Обработка дисбаланса: рецепты

  • Веса классов:
    • LGBM: scale_pos_weight ≈ N_neg/N_pos; проверить устойчивость OOF.
    • CatBoost: class_weights=[w0,w1]; часто + легкая L2.
    • XGB: scale_pos_weight + max_delta_step при сильном дисбалансе.
  • Focal loss (γ=1–2, α для «1»): уменьшает вклад легко-классифицируемых «0», усиливает тяжёлые примеры.
  • Down-/Up-sampling:
    • Down-sample негатива до соотношения 1:10..1:20 → быстрее обучение.
    • Корректировать пороги/калибровку (иначе «съедится» базовая вероятность).
    • Up-sample позитива осторожно (seed-bagging лучше, чем дубли).
  • Threshold moving: оптимизация порога на OOF под бизнес-метрику (cost-sensitive).
  • Cost-sensitive fit: sample_weight на объект (дорогие FN/FP).

6) Калибровка вероятностей

  • Platt scaling (логистическая регрессия поверх OOF-скор): быстро, часто достаточно.
  • Isotonic regression — мощнее, но может переобучать; контролировать по фолдам.
  • Оценка: Brier, ECE, релайбилити-плоты.
  • Правило: калибровку подгоняем на OOF, применяем к test-скор.

7) CV под скоринг и фрод

  • Скоринг (panel с клиентами): GroupKFold(client_id); если есть время — rolling по месяцам с удержанием клиента.
  • Фрод (транзакции): TimeSeriesSplit/PurgedKFold + gap; признаки только из прошлого; кодировки — rolling/expanding.
  • Adversarial validation: если drift висок, усиливаем time-aware, сокращаем «токсичные» фичи.

8) Безопасный фичинжиниринг (короткий каталог)

Персональные/панельные (скоринг):

  • Aggregates per client: mean/median/max/min/std, доли просрочек, DPD суммарные/макс, число активных продуктов.
  • Тренды (slope/Δ) по лимитам/балансам/расходам, recency (последняя активность).
  • Стабильность дохода/платежа (коэфф. вариации), utilization (balance/limit), category mix.

Транзакции/фрод:

  • RFM (recency/frequency/monetary), velocity (операций/минуту), суммы по окнам 1/7/30/90.
  • Перекрёстные частоты: user×merchant, card×device, ip×hour, rare pooling.
  • Time-aware кодировки (rolling counts/expanding mean), decay-агрегаты (экспон. спад).
  • Device/Geo сигнатуры: расстояния/переключения, часовое распределение.
Правило: для всех агрегатов — никакого будущего; для кодировок — только OOF/time-aware.

9) Базовые пресеты моделей

  • CatBoost: сильный старт с категориальными; class_weights, depth 6–8, l2_leaf_reg 3–10, auto_class_weights=None/Balance.
  • LightGBM: num_leaves 63–255, feature_fraction 0.6–0.9, bagging_fraction 0.6–0.9, scale_pos_weight, min_data_in_leaf ↑ при шуме.
  • XGBoost: max_depth 6–10, eta 0.02–0.1, subsample/colsample 0.6–0.9, scale_pos_weight, eval_metric=auc/pr-auc.

Ансамбль (30–45 мин): 2–3 модели, веса по OOF (Optuna/по сетке), bagging seeds (3–5), усреднение логитов.

10) Мини-практикум A (60–90 мин): скоринг (panel)

Цель: устойчивый ранжирующий скор (AUC/AMEX-стайл), без фиксации порога.

План:

  1. GroupKFold(client_id) + CatBoost-бейзлайн → OOF AUC.
  1. Доб. 5–8 безопасных агрегатов per client → ΔOOF.
  1. Лёгкий бленд CatBoost+LGBM (веса по OOF).
  1. Проверка drift (AV), отчёт по стабильности.

Артефакты: OOF-лог, cv.yaml, featureset_v1.json, сабмит_1–2.

11) Мини-практикум B (60–90 мин): фрод (transactions)

Цель: PR-AUC/Recall@K, time-aware фичи.

План:

  1. TimeSeriesSplit(gap) + LGBM с scale_pos_weight.
  1. Rolling counts (user/item 7/30), velocity, rare pooling → ΔOOF (PR-AUC).
  1. Threshold moving под Recall@K на OOF.
  1. (Опционально) focal-loss/alpha-BCE и сравнить.

Артефакты: калибровка/порог, метрики PR-AUC/Recall@K, сабмит_финал.

12) Домашка-разминка (до спринта)

  • Реализовать утилиты: compute_class_weights, apply_downsample, fit_calibrator, optimize_threshold(metric_fn).
  • Сформировать бизнес-метрику (таблица издержек FP/FN или Capture@K) и функцию её расчёта на OOF.

13) Самостоятельный 5-часовой спринт

Выбор кейса (преподаватель выдаёт):

  • Скоринг (мульти-стейтменты на клиента) или Фрод (поток транзакций).

    Требования:

  • Корректная CV (Group/Time/Purged), минимум 8 безопасных фич, один «смелый» приём + стресс-тест.
  • Один сабмит «чистый ранжирующий», один — с калиброванным порогом под бизнес-метрику.
  • Отчёт: CV↔LB, дрифт/AV, выбор обработки дисбаланса, паспорт решения (1 страница).

14) Чек-лист утечек (версия М3)

  • Кодировки/агрегаты — только OOF и/или только из прошлого (embargo).
  • Группы клиентов не пересекаются между train/valid.
  • Threshold/порог подбираем по OOF, не по публичному LB.
  • Любая «магия» проверяется переносом на альтернативный hold-out.

15) Оценивание

  • Корректность (35%): сплит и time-политика, отсутствие утечек.
  • Результат (30%): ΔOOF/ΔLB, устойчивость (std по фолдам), адекватность top-K.
  • Инженерия (20%): воспроизводимость, скорость (лог таймингов), аккуратные артефакты (OOF/кал. модель).
  • Отчёт (15%): ясный выбор техники дисбаланса/метрики/порога.

16) Что уходит «в рюкзак»

  • imbalance.py (веса, сэмплинг, focal-loss обёртки), calibration.py (Platt/Isotonic), thresholds.py.
  • Шаблон scoring_feature_factory.py (panel) и fraud_feature_factory.py (time-aware).
  • Пресеты lgbm_credit.yaml, catboost_fraud.yaml + скрипт бленда логитов.
  • Чек-листы: «дисбаланс за 15 минут», «калибровка», «Recall@K/Top-K лайфхаки».

17) Встроено в 5-часовой формат (ритм)

  • 0–30 мин: CV (Group/Time) + бейзлайн с весами → OOF/сабмит_1.
  • 30–90 мин: безопасные фичи (агрегаты/rolling/rare pooling) → ΔOOF/сабмит_2.
  • 90–150 мин: обработка дисбаланса (focal/досэмплинг), стабилизация.
  • 150–210 мин: калибровка и/или threshold moving под бизнес-метрику.
  • 210–240 мин: лёгкий ансамбль, финальный сабмит + паспорт решения.

Готов продолжить и детализировать кодовые скелеты (imbalance.py, calibration.py, фабрики фич) ровно под твой репозиторий и стиль запуска.
