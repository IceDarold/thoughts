М16. MLOps-лайт: эксперименты, артефакты, упаковка и «одна кнопка»

ниже — подробный Модуль 16. Фокус: MLOps-лайт для соревнований — как зафиксировать эксперименты, версии данных и артефакты, собрать “одной кнопкой”, быстро генерить отчёты и паковать сабмиты/датасеты под Kaggle/олимп 5-часовой формат.

1) Цели модуля

  • Стабилизировать воспроизводимость: конфиги, сиды, фикс сплитов, подписи данных/фич.
  • Вести эксперименты как журнал: run-id, метки, сравнение, автосбор отчётов.
  • Стандартизовать артефакты: OOF/test предикты, fold-map, фич-кэш, сабмиты, метрики, профили времени/памяти.
  • Настроить “one-click” пайплайн (build→train→oof→test→blend→submit) и аварийные режимы.
  • Научиться паковать решение: Kaggle API (datasets/submissions), offline-бандлы для экзамена.

Выход: папка ops/ + experiments/ с готовыми скриптами, шаблонами отчётов и чек-листами. Любую модель из модулей М2–М15 можно прогнать end-to-end и воспроизвести.

2) Формат и тайминг

  • 2 занятия × 2 часа (теория + лайв-код).
  • Самостоятельный мини-спринт 3–4 часа (оцифровать свой текущий проект).
  • Итог: «паспорт пайплайна» (1 стр) + пара сабмитов из «одной кнопки».

3) Карта решений за ≤10 минут

  1. Источник правды — один task.yaml: пути к данным, ключи id/time/group, метрика, тип CV.
  1. Сигнатуры — data_signature.json (schema+nrows+head-hash), fold_map.parquet заморожен.
  1. Артефакты — строгая схема папок; сабмиты/OOF хранятся вместе с метаданными.
  1. Запуск — python -m ops.runner --preset fast|standard|max --config configs/task.yaml.

4) Стандарт папок и артефактов

.

│   ├── model_*.yaml            # пресеты моделей
│   └── blend.yaml              # правила бленда (М12)
├── data/                       # только чтение (symlink/readonly)
├── experiments/                # журнал экспериментов

│       ├── params.yaml         # слепок конфигов
│       ├── data_signature.json
│       ├── timings.json        # профили (М13)
│       ├── memory.json
│       ├── cv_report.json      # метрики по фолдам
│       ├── oof.parquet         # OOF предсказания
│       ├── test.parquet        # test предсказания (по моделям)
│       ├── submit.csv          # сабмит
│       └── report.md           # автоотчёт
├── features/                   # фабрика фич (М11) + кэш
│   └── feature_store/...
├── models/                     # train/predict, сериализация best params

5) Конфиг-политика

  • Один task.yaml описывает задачу (ключи, метрика, сплит, пути).
  • Модели — отдельные model_*.yaml (гиперы, лоссы, ранние стопы).
  • Бленды — blend.yaml (список OOF/test путей, стратегия: mean/logit/NNLS).
  • Любой запуск сохраняет слепок конфигов в experiments/<run_id>/params.yaml.

6) Сиды, детерминизм, версии

  • Фиксируем PYTHONHASHSEED, NumPy/Random/LightGBM/Cat/XGB сиды.
  • Версии библиотек — лок-файл requirements.lock.
  • Fold-map неизменяемый; смена сплита = новый task.yaml с новым id сплита.
  • Проверка детерминизма: повторный запуск → ΔOOF ≤ 1e-6 (тест в CI-лайт).

7) Журнал экспериментов (без тяжёлых платформ)

  • experiments/ — одна папка на запуск: run-id = дата_время-краткое_имя.
  • Автоматически сохраняем: метрики (по фолдам/OOF), профили времени/памяти, ссылки на фич-кэш, список файлов сабмита.
  • Скрипт make_report.py → report.md с таблицами:
    • CV по фолдам (+std), важности фич, Δ к базовой, время/память, CV↔LB трекер.

(хочешь — подключишь W&B/MLflow поверх; интерфейсы оставляем совместимыми)

8) Артефакты и схемы данных

  • OOF: колонки id, fold, y_true, y_pred, (для @K — group_id).
  • Test: id, y_pred (или rank/score для ранжирования).
  • CV-отчёт: per-fold метрика, среднее, std, доверительные интервалы.
  • Сабмит: строгий schema.json (проверяем порядок id/столбцы перед отправкой).
  • Signatures: data_signature.json, хеши фич-кэша (см. М11).

9) One-click runner: сценарии

  • -preset fast|standard|max (см. М13).
  • -stage: fe|train|oof|test|blend|submit|all.
  • Грейсфул-деградация: при риске не уложиться во время → автоматически переключаемся на fast-профиль и лёгкий бленд.
  • Fail-fast: любой чек (id-порядок, schema, memory budget, leak-guard) падает с понятным сообщением.

10) Качество до сабмита (pre-flight checks)

  • No-leak (М11/М15): OOF-генерация/TS-gap/embargo.
  • Schema-guard: названия/типы колонок, id-порядок совпадает с эталоном.
  • CV↔LB sanity: есть хотя бы один “якорный” сабмит с внятным CV.
  • Память/время: memguard и timings зелёные; сабмит ≤ N МБ.
  • Документация: краткий README_submit.md (метрика, сплит, ключевые фичи, риски).

11) Упаковка под Kaggle/олимп

  • Kaggle API: ops/submit.py умеет:
    • собрать submit.csv + README_submit.md;
    • (опц.) упаковать Kaggle Dataset с OOF/фич-кэшем/weight’ами;
    • отправить сабмит и сохранить submission_id в experiments/<run_id>/.
  • Offline-бандл (олимп/экзамен без интернета):
    • bundle.zip = runner.py + configs/ + models/ + минимальный feature_store/ + Makefile.
    • Тестируем локально: make all → сабмит в outputs/submit/.

12) Быстрая документация и отчёты

  • report.md (авто):
    • “что запускалось”, метрики, сравнение с базой, профили времени/памяти, важности.
    • “паспорт решения” (1 страница): CV-схема, метрика, фич-ядро, риски/AV, план финалки (safe/bold).
  • cv_lb_tracker.csv: история сабмитов (время, run-id, CV, LB, сплит, заметки).

13) Мини-практикумы

A) Оцифруй проект (60–90 мин)

  1. Перевести текущий код на task.yaml + runner.py.
  1. Зафиксировать fold_map.parquet, data_signature.json.
  1. Прогнать -preset fast → получить первый report.md.

B) Автосабмит и отчёт (60–90 мин)

  1. Сделать два запуска (baseline и улучшенный).
  1. submit.py отправляет оба; cv_lb_tracker.csv фиксирует результаты.
  1. Сгенерировать сравнение (Δметрики, время/память).

C) Аварийный режим (45–60 мин)

  1. Смоделировать ограничение времени/памяти → переключиться на fast.
  1. Проверить, что пайплайн выдаёт валидный сабмит и отчёт ≤ лимита.

14) Самостоятельный мини-спринт (3–4 часа)

Задание: упаковать одно из решений (М2–М12) под MLOps-лайт.

Требования:

  • Один task.yaml, fold_map.parquet, data_signature.json.
  • Два запуска: fast и standard; оба с валидными report.md.
  • cv_lb_tracker.csv с 2–3 сабмитами; один offline-bundle.zip.

15) Чек-лист (версия М16)

  • Повторный запуск → те же OOF/метрики (±1e-6).
  • Все артефакты на месте (OOF/test/submit/отчёты/подписи).
  • Нет утечек; schema/id-порядок проверены.
  • “Одна кнопка” работает в fast и standard.
  • Есть bundle.zip для офлайна и README_submit.md.

16) Оценивание

  • Корректность (35%): воспроизводимость, no-leak, валидные артефакты.
  • Результат (30%): два сабмита из runner’а, понятная корреляция CV↔LB.
  • Инженерия (20%): чистота репо, автоматические отчёты, аварийный режим.
  • Отчёт (15%): паспорт пайплайна, трекер сабмитов, выводы.

17) Что уходит «в рюкзак»

  • ops/runner.py, ops/submit.py, ops/registry.py, ops/seed_guard.py, ops/io_utils.py.
  • Шаблоны configs/task.yaml, blend.yaml, README_submit.md, report.md.
  • Скрипт make_report.py, трекер cv_lb_tracker.csv (шаблон).
  • Инструкции bundle_checklist.md и panic_plan.md.

18) Встроено в 5-часовой формат (ритм)

  • 0–30 мин: зафиксировать task.yaml, сигнатуры, поднять runner fast → сабмит_1.
  • 30–90 мин: подключить фич-кэш/ансамбль из прошлых модулей, runner standard → сабмит_2.
  • 90–150 мин: довести отчёт/трекер, проверить offline-бандл.
  • 150–210 мин: автосабмит + sanity-checks (schema/no-leak/детерминизм).
  • 210–240 мин: финальный паспорт пайплайна, список «риск-точек» и как их чинить за 15 минут до дедлайна.

Если хочешь, под твой текущий темплейт накину минимальный ops/runner.py + task.yaml и заготовки отчётов — вставишь модели из предыдущих модулей и поедешь на автоколёсах.
