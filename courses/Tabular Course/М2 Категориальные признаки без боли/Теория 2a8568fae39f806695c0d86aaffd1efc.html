Теория

ниже — «полная теория» для М2. Это не туториал, а опорный конспект: определения, формулы, анти-утечки, когда что работает и почему. В конце — боевой чек-лист под 5-часовой спринт.

1) Что такое «категориальные» (и почему это сложно)

Типы:

  • Номинальные (color, city): нет порядка.
  • Порядковые / ранговые (education_level): порядок есть.
  • Высокая кардинальность (user_id, item_id, merchant_id).
  • Мультизначные (tags = {“sport”, “kids”}).
  • Иерархические (city→region→country).

Риски:

  • Взрыв размерности (one-hot).
  • Утечки при таргет-энкодингах.
  • Неизвестные/редкие категории на тесте.
  • Спуриа (случайные совпадения в высоких картах).

2) Зависимость от модели (что «ест» ваша модель)

  • Деревья/GBDT (LightGBM/XGBoost/CatBoost):
    • CatBoost имеет нативные CTR/ordered-статистики → лучший выбор для high-card c минимальной болью.
    • LightGBM умеет categorical_feature: сортирует категории по статистике и режет по порядку (без one-hot); безопасно при честном CV.
    • XGBoost: чаще используют one-hot/hash/target. (Если есть нативные cats — всё равно проверяйте CV.)
  • Линейные/логистические/GLM: нужны численные кодировки без нелинейной магии: one-hot, WOE, таргет-энкодинг с сильным сглаживанием.
  • NN/табличные трансформеры: entity-эмбеддинги + нормальные числовые фичи.

3) Базовые кодировки

3.1 One-Hot (OHE)

  • (x \in {a_1,\dots,a_K} \Rightarrow \mathbf{1}[x=a_k]).
  • Плюсы: простая, честная, интерпретируемая, отлично с линейками.
  • Минусы: при (K\gg 10^3) взрывает память; редкие категории бесполезны.
  • Приёмы: top-K + __RARE__, отдельная категория __UNK__ для новых значений.

3.2 Частоты/счёты (Frequency/Count)

  • (\mathrm{freq}(c)=\frac{n_c}{N}), (\mathrm{count}(c)=n_c).
  • Плюсы: дёшево, всегда полезно деревьям, нулевая утечка.
  • Минусы: не учитывает таргет.

3.3 Хеш-трик (Feature Hashing)

  • (h(c)\in{1,\dots,d}), иногда со знаком ((\pm1)).
  • Плюсы: фиксированный размер, быстрый, защитный от «новых» категорий.
  • Минусы: коллизии (смещение/шум); выбирайте (d) из бюджета памяти, типично (2^{12}\dots2^{18}).

3.4 Бинарная/Хельмерт/Суммовая

  • Компактные кодировки для средних K (бинарная ≈ (\lceil \log_2 K\rceil)).
  • Полезно для линейных, но обычно проигрывают таргет-кодировкам.

4) Таргет-кодировки (mean/likelihood/WOE) — теория и защита

4.1 Mean/Target Encoding (регрессия/классификация)

Идея: заменить категорию на ожидаемое значение таргета для этой категории.

M-estimate (сглаживание):

[

\mathrm{TE}(c) ;=; \frac{\sum_{i: x_i=c} y_i + m\cdot \mu}{n_c + m}

]

где (\mu) — глобальное среднее, (m) — «сила» приора (сколько «виртуальных наблюдений»).

Бета-сглаживание (для Bernoulli (y)):

[

\mathrm{TE}(c) ;=; \frac{\alpha + \sum y}{\alpha + \beta + n_c}

]

(\alpha,\beta) — параметры приора.

Для регрессии с heavy-tail: используйте Huber-среднее или winsorize.

Ключ: избегать утечки. Нельзя считать TE по всем данным и затем обучаться на этих значениях — модель «видит» таргет.

Три безопасных режима:

  • K-Fold OOF: для каждого фолда TE считается только на train-части; применяем к валу этого фолда.
  • Leave-One-Out (LOO): (\frac{\sum y - y_i}{n_c-1}); добавляйте шум (\mathcal{N}(0,\sigma)) или (\times(1+\epsilon)). Работает, но при малых (n_c) шумнее.
  • Ordered/Expanding (время): сортируем по времени/перестановке и считаем expanding-mean из прошлого.

Смягчение утечки:

  • Шум (Gaussian/Multiplicative) на TE в train.
  • Ограничить влияние редких (порог (n_c<r \Rightarrow) __RARE__).
  • Для multi-class: либо K столбцов TE (per class), либо response coding (логиты классов).

4.2 WOE (Weight of Evidence) — бинарная классификация

[

\mathrm{WOE}(c)=\ln\frac{\Pr(c \mid y=1)}{\Pr(c \mid y=0)} ;=; \ln \frac{\frac{n_{c,1}}{N_1}}{\frac{n_{c,0}}{N_0}}

]

  • Плюсы: линейные модели получают монотонный сигнал, хорошо «садится» в логистику, интерпретируемый, устойчив к дисбалансу.
  • Минусы: требует OOF/ordered расчёта; для малых (n_{c,\cdot}) — сглаживание (аддитивная лапласовская постоянная).

4.3 GLMM-encoding (random effects)

  • Считайте категорию как случайный эффект (u_c \sim \mathcal{N}(0,\tau^2)) в регрессии (y = X\beta + u_c + \epsilon).
  • На практике — «бейзийский» TE с shrinkage к глобали. Делается через OOF-оценку эффекта; даёт аккуратное сглаживание при высоких картах.

5) Время, группы и честная валидация (главный источник боли)

Правило: где таргет участвует в расчёте признака — обязателен OOF/time-aware протокол.

  • GroupKFold: не делим одну сущность (user_id/merchant_id) между train/valid при генерации TE.
  • Time/Purged/Embargo: если есть время, используем rolling/expanding, gap ≥ max_lag.
  • Стабильность: фиксируйте fold_map.parquet; любые TE пересчитываются по нему.

6) Редкие/невиданные категории

Стратегии:

  • Top-K + __RARE__: всё, что реже порога — в одну корзину.
  • Поисправленная TE: при (n_c < r) используем более сильный приор (большой (m)).
  • Иерархический бек-офф: city→region→country: если нет сигнала на city — берём TE(region), затем TE(country), затем (\mu).
  • __UNK__ для новых на тесте: код как (\mu) или TE родительского уровня.

7) Перекрёстия (feature crosses) и мультимодальные cats

  • A×B: полезно, но экспоненциально растёт. Спасает:
    • ограничение на топ-комбинации;
    • hashing пар (двойной хеш: (h(a,b)=h_1(a)\oplus h_2(b))).
  • Мультизначные (tags): bag-of-categories или TF-IDF по словарю тегов; для деревьев — сумма/среднее TE по тегам.

8) Эмбеддинги категорий (entity embeddings)

  • Для категории с (K) уникальных значений обучаем эмбеддинг (\mathbf{e}_c \in \mathbb{R}^d).
  • Эвристика для размерности: (d \in [\min(50,\lfloor 1.6\cdot K^{0.56}\rfloor), \min(100, \lfloor \sqrt{K}\rfloor)]); на практике 8–32 часто достаточно.
  • Обучаем простой MLP на задаче, берём эмбеддинги как фичи для GBDT. Обязательно OOF-извлечение или отдельный fit на train-окне каждого фолда.
  • Плюсы: компактно, ловит неявные сходства; Минусы: нужен NN-контур и дисциплина OOF.

9) Специфика фреймворков

CatBoost

  • «Ordered boosting»: считает CTR/TE по префиксу перестановки — встроенная защита от утечек.
  • Параметры: one_hot_max_size (мелкие — OHE), max_ctr_complexity (перекрёстия), l2_leaf_reg ↑ для устойчивости.
  • Практика: не делайте внешний TE — доверьтесь CatBoost, добавляйте только частоты/счёты/иерархический бек-офф как независимые фичи.

LightGBM

  • Указывайте categorical_feature и не превращайте в произвольные int-коды без флага — иначе модель введёт ложный порядок.
  • При очень больших K: freq/hash/TE(O O F) + оставить часть полей как categorical_feature с max_cat_threshold.

XGBoost

  • Если без нативных cats: OHE/freq/hash/TE(OOF). Следите за шириной матрицы (sparse CSR помогает).
  • Регуляризация: min_child_weight и lambda выше для TE-фич.

Линейные/GLM

  • OHE/WOE/TE(сильное сглаживание). Для WOE можно добавить монотонные ограничения во внешних методах (или биннинг + сплайн).

10) Диагностика и стабильность

  • Пермутационная важность по фолдам: TE-фичи не должны давать «чудес» на одном фолде и провал на другом.
  • AV/PSI (train vs valid/test) по TE-фичам: резкий сдвиг — признак подгона.
  • Leak-unit-tests: любой TE должен падать, если попробовать «fit на всём train и применить на нём же».

11) Типовые рецепты (когда что брать)

  • High-card ID + деревья: CatBoost (нативно) или LGBM с freq/count + OOF-TE (m-estimate) + hash-cross A×B.
  • Линейки: OHE для малых K; WOE/TE(OOF) для больших; обязательно регуляризация и контроль коллинеарности.
  • Time-series спрос/продажи: ordered/expanding TE только из прошлого, с decay-весами.
  • Фрод/скоры с дисбалансом: WOE (биннинг по стабильным границам) + OOF-TE; оценка по PR-AUC.
  • Recsys (user_id, item_id): эмбеддинги (OOF-извлечение) + freq/recency; для re-rank — TE на item×user-срезах, но строго OOF.

12) Формулы/детали, о которых часто спрашивают

Шум к TE (уменьшить переобучение):

[

\tilde{v} ;=; v \cdot (1 + \epsilon), \quad \epsilon \sim \mathcal{N}(0,\sigma^2);; \text{или};; \tilde{v}=v+\epsilon

]

Типично (\sigma\in[0.01, 0.05]).

LOO-исправления на малых (n_c):

[

\mathrm{TE}_{\text{LOO}}(i) ;=; \frac{S_c - y_i + m\mu}{(n_c - 1) + m}

]

Иерархический бек-офф (рекурсивно):

[

\mathrm{TE}(city) \leftarrow

\begin{cases}

\mathrm{TE}{\text{own}}, & n{city}\ge r\

\lambda \cdot \mathrm{TE}(region) + (1-\lambda)\cdot \mu, & n_{city}<r

\end{cases}

]

TF-IDF для мультизначных cats:

[

\mathrm{tfidf}(t,i) = \mathrm{tf}(t,i)\cdot \log\frac{N}{df(t)}

]

где «терм» — тег, «документ» — объект/сессия.

13) Типичные ошибки

  • Присвоили категориям произвольные int-коды и отдали линейной модели → ложний порядок.
  • Посчитали TE на всём train и обучились на нём же.
  • Смешали группы (user_id) между train/valid при TE.
  • Положили редкие категории в OHE → 99% нулей, overfit по шуму.
  • Забыли __UNK__/__RARE__ → провал на тесте.
  • Не зафиксировали fold_map → TE «плавает» между перезапусками.

14) Боевой чек-лист под 5-часовой спринт

Выбор:

  1. Модель деревьев? → CatBoost (нативно) или LGBM + categorical_feature.
  1. Карта > 5k? → freq/count + OOF-TE (m-estimate), top-K + __RARE__/__UNK__, hash для A×B.
  1. Есть время? → добавьте 1–2 эмбеддинга (OOF-извлечение) для самых больших ID.

Анти-утечки:

  • Зафиксируйте сплит (Group/Time/Purged).
  • TE — только OOF/ordered; для TS — expanding с gap.
  • Добавьте шум и сильное сглаживание для редких.

Устойчивость:

  • Пермутационная важность по фолдам, AV/PSI на TE-фичах.
  • Альт-сплит: TE должны держать знак/вклад.

Независимые фичи (всегда можно успеть):

  • freq/count, доли по группе, длина множества тегов, «новизна»/возраст категории, иерархический уровень.

15) Мини-таблица «когда что взять»

  • Малый K (≤30): OHE (+ порядковая кодировка, если реально порядок), всё просто.
  • Средний K (30–1000): OHE для топ-K, остальное __RARE__ + freq; при линейках — WOE/TE(OOF).
  • Высокий K (≥1000): CatBoost нативно или freq + OOF-TE + hash-cross; эмбеддинги при наличии NN-контуров.
  • Мультизначные: TF-IDF/сумма TE по тегам + hash-bag.
  • Time-series: только ordered/expanding TE с gap.

16) Что фиксировать в отчёте (перед сабмитом)

  • Какой тип CV/сплит и как считались TE (K-Fold OOF / LOO / Ordered).
  • Значения (m), (\alpha,\beta), (\sigma) шума; пороги rare/top-K.
  • Наличие __UNK__/__RARE__ и иерархического бек-оффа.
  • Проверки AV/PSI на TE-фичах и стабильность на альтернативном сплите.

Если нужно, могу к этой теории быстро приложить два «боевых» рецепта:

а) минимальный OOF-таргет-энкодер (m-estimate + шум) с поддержкой Group/Time;

б) шаблон иерархического бек-оффа (city→region→country) — под твой пайплайн.
