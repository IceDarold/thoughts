Теория

ниже — «полная теория» для М3. Это опорный конспект: постановка, утечки/смещения, метрики, дисбаланс, скоркарды (WOE/IV), калибровка/порог, reject inference, стабильность (OOT/PSI), монотонности, и быстрый чек-лист под 5-часовой спринт.

1) Постановка задачи и таргет

Цель: оценить вероятность дефолта (PD) заявителя на горизонте (H) (обычно 6–12 мес) по информации на момент решения.

Таргет (двойка):

(y=1) ⇔ наступило событие default (напр., 90+ DPD в течение (H) месяцев от активации); (y=0) — иначе. Важно задать:

  • Observation window (входные данные до (t_0), даты решения);
  • Performance window (H) (наблюдаем дефолт после (t_0));
  • Censoring: заявки ближе к концу среза не успевают прожить (H) → исключать/взвешивать.

Группы: один customer_id не должен пересекать train/valid. Для повторных заявок: либо «первая», либо time-aware CV.

2) Анти-утечки (главное)

Запрещены фичи, известные после (t_0): статусы погашения, суммы взысканий, апдейты бюро после заявки, признаки «получил/не получил кредит».

Time/Purged CV: валидные окна отстают по времени (embargo ≥ макс-лаг/окно).

GroupKFold: разделение по клиенту/домохозяйству/работодателю.

Feature fit (нормировки, таргет/WOE) — только OOF/ordered/time-aware.

3) Смещения данных (selection bias)

Accepted-only bias: в исторических данных известны исходы только по одобренным; отклонённые не наблюдаем → выборка не репрезентативна.

Reject inference (см. §12): корректируем смещение оценивая «вероятность одобрения» и/или «дорисовывая» таргеты для отклонённых. На спринте — минимум: оценка переносимости на OOT без RI, аккуратный cut-off по бизнес-констрейнтам.

4) Метрики качества

  • AUC ROC, KS (макс. разность CDF «good/bad» по скору), Gini = 2·AUC−1 — ранжирование.
  • PR-AUC — информативно при низком bad-rate.
  • Brier/ECE — калибровка вероятностей.
  • Лифт/Гейнс по децилям — бизнес-читаемо.
  • Экономика: (\mathbb{E}[\text{loss}] = \text{PD}\times \text{LGD}\times \text{EAD}). Для отбора порога — матрица издержек.

Регуляторный стандарт: фиксируем OOT-сплит (out-of-time) на отдельном периоде; мониторим PSI (см. §11).

5) Дисбаланс классов (bad-rate ≪ 1)

5.1 Веса vs ресемплинг

  • Class weights в лоссе (LGBM/XGB/Cat: scale_pos_weight≈Nneg/Npos, auto_class_weights=Balanced). Сохраняет калибровку.
  • Undersampling негатива — прост, теряем информацию.
  • Oversampling/SMOTE — осторожно: переобучение, ломает калибровку деревьев.

    Рекомендация: сначала веса, оценка по исходному распределению; ресемплинг — как эксп. сравнение.

5.2 Лоссы под дисбаланс

  • Logloss + class weights — база.
  • Focal loss: (-\alpha(1-p)^\gamma y\log p - (1-\alpha)p^\gamma (1-y)\log(1-p)) — фокус на трудных примерах.
  • Для деревьев — через кастомную цель или имитацию весами.

6) Модели и ограничения

  • Логистическая регрессия (scorecard-классика): интерпретируемо, легко калибровать, даёт «reason codes»; любит WOE/биннинг и монотонные связи.
  • GBDT (Cat/LGBM/XGB): сильнее по AUC/KS, гибкие; для регулятора — монотонные constraints и reason-коды (SHAP + биннинг) усиливают объяснимость.
  • Fair/Compliance: не использовать защищённые признаки; делать slice-оценки (см. §11) — но это отдельная тема.

7) Скоркард: биннинг, WOE, IV

7.1 Биннинг и WOE

  • Разбиваем числовой признак на бины (квантиль/ChiMerge/MDL).
  • WOE для бина (b):

    [

    \mathrm{WOE}_b=\ln\frac{\text{Bad}b/\text{Bad}{\text{all}}}{\text{Good}b/\text{Good}{\text{all}}}

    ]

    (плюс сглаживание (\epsilon) к счётчикам).

  • Пропуски — отдельный бин.
  • Добиваемся монотонной зависимости WOE по смысловой оси (дообъединяем бины, isotonic-merge).

7.2 Information Value (IV)

[

\mathrm{IV} = \sum_b (\text{DistrBad}_b - \text{DistrGood}_b)\cdot \mathrm{WOE}_b

]

Эвристика: <0.02 — слабый; 0.02–0.1 — умеренный; 0.1–0.3 — хороший; >0.3 — сильный (иногда подозрительно сильный).

Важное: IV — до-модельная эвристика, не заменяет OOF-валидацию.

7.3 Логистическая на WOE

Формально:

[

\log\frac{p}{1-p} = \beta_0 + \sum_j \beta_j ,\mathrm{WOE}_j(x)

]

даёт стабильную монотонность и «reason codes» (вклад бина). L1/L2 — для отбора.

8) Признаки кредитного риска (без утечки)

  • Бюро/история: количество/доля просрочек (30/60/90) за окна (12/24 мес), «время с последней просрочки», inquiries за 3/6/12 мес.
  • Дебт-бурден: utilization (баланс/лимит), DTI (debt-to-income), число активных счетов.
  • Стабильность: стаж на работе, адрес/телефон срок, «частота смен».
  • Поведение по счёту (для behavior-скоринга): payment-ratio, min-pay history, overlimit flags — только до (t_0).
  • Иерархии/бэкофф: индикаторы отрасли работодателя → сектор → регион.

    Все агрегаты time-aware, окна не пересекают валид.

9) Калибровка и «score-to-odds»

Калибровка (на OOF): Platt (логистическая) / Isotonic → снижает смещение PD.

Scorecard-скейл:

[

\text{Score} = \text{Offset} + \text{Factor}\cdot \ln\frac{1-p}{p}, \quad

\text{Factor} = \frac{\text{PDO}}{\ln 2}

]

где PDO — прирост очков при удвоении odds (напр., PDO=20). Offset задаём из «базовых odds» (например, 600 очков при odds=50:1).

Калиброванный PD (\rightarrow) cut-off по бизнес-ограничениям (целевой PD портфеля, доля одобрений).

10) Порог и экономический выбор

Пусть прибыль от «good-approved» = (G), потери от «bad-approved» = (C). Для кандидата со скором (p=\text{PD}):

[

\mathbb{E}[\text{profit}] = (1-p)\cdot G - p\cdot C

]

Одобрим если ( \mathbb{E}[\text{profit}]>0 \Rightarrow p < \frac{G}{G+C}).

На практике:

  • Constraint-based: выбрать порог, чтобы доля одобрений и средний PD в принятом слое соответствовали целям.
  • Decile-анализ: таблица LR (loss ratio) по бинам — проверка здравости cut-off.

11) Стабильность и мониторинг

  • OOT (out-of-time) валидация — главный тест.
  • PSI по скору/ключевым фичам (сравнение train vs OOT). Правила: <0.1 — норм, 0.1–0.25 — внимание, >0.25 — сдвиг.
  • Slice-оценки: возрастные группы, регионы, продукты, каналы → контролируем AUC/KS/PD drift.
  • KS drift: сравнить KS в dev vs OOT; падение → переобучение/смещение.
  • Back-testing PD: калибровка по децилям — факт против ожидаемого.

12) Reject Inference (когда и как)

Нужно, если want: портфельная PD после cut-off и устойчивость меняются из-за политики одобрения.

  • Propensity weighting: модель (q(x)=\Pr(\text{approved}\mid x)) → веса (w=1/q(x)) при обучении PD.
  • Parceling: разметить rejected, присвоив им (y) по их скору/децилю (консервативный сценарий).
  • EM/Heckman-подходы: совместное моделирование «одобрение↔исход», дороже.

    Риски: внесение моделирующего допущения. В спринте — как минимум оценить влияние (с/без RI) на OOT.

13) Монолитные деревья vs скоркард

  • Скоркард: +интерпретируемость, +контроль монотонности, +стабильность; − потолок AUC.
  • GBDT: +качество, +нелинейности/взаимодействия; вводим монотонные constraints (LGBM/XGB), глубину ≤6–8, min_data_in_leaf↑, lambda_l2↑ → меньше дрожи.

    Компромисс: дерево-база + табличка «reason-codes» через биннинг/SHAP-группы.

14) Фейрнес и регуляторика (кратко)

  • Исключать защищённые признаки; делать disparate impact/slice-метрики; хранить reason codes (топ-вклады фичи в отказ).
  • Документировать: постановку таргета, окна, сплиты, PSI/KS/калибровку, выбор порога/политики.

15) Типовые пайпы (когда что)

  • Низкий bad-rate (<3%), быстрый запуск: Cat/LGBM с scale_pos_weight, калибровка, decile-лифт, cut-off по PD-целям.
  • Регуляторная интерпретация: скоркард: биннинг→WOE/IV→логистическая→score-to-odds; монотонности и reason-codes.
  • Behavior-скоринг (портфель): time-aware окна, лаги платежей, осторожно с «будущими» фичами; OOT по месяцам.
  • Мультипродукт: отдельные модели или мультиклассовый слой «product» как фича (без утечки).

16) Частые ошибки

  • Микс customer_id между train/valid; leakage из post-decision полей.
  • TE/WOE посчитаны «на всём train» и применены там же.
  • Оценка на ресемплированном дистрибутиве (некорректные метрики).
  • Cut-off выбран «по AUC» без экономики/ограничений портфеля.
  • Нет OOT; нет PSI/decile back-test; порог «плавает» по времени.

17) Формулы и полезные куски

  • KS: (\max_t (\mathrm{TPR}(t)-\mathrm{FPR}(t))).
  • Gini: (2\cdot\mathrm{AUC}-1).
  • WOE smoothing: (\ln\frac{(\text{Bad}_b+\epsilon)/(\text{Bad}+\epsilon B)}{(\text{Good}_b+\epsilon)/(\text{Good}+\epsilon B)}).
  • Score scale: ( \text{Score}=A - B\cdot \ln\frac{p}{1-p}), (B=\frac{\text{PDO}}{\ln 2}).
  • Focal: см. §5.2.

18) Боевой чек-лист (5-часовой спринт)

0) Сетка валидации

  • Зафиксировать OOT (последние месяцы) + GroupKFold(customer) в dev, embargo по времени.

1) База

  • LGBM/Cat с scale_pos_weight (≈neg/pos), простое фич-ядро (история просрочек, utilization, inquiries, стаж).
  • Метрики: AUC/KS, PR-AUC; отчёт по децилям (лифт, PD).

2) Интерпретация/устойчивость

  • Биннинг ключевых фич → WOE sanity (монотонность).
  • Пермутационная важность по фолдам; OOT-чек.

3) Калибровка и cut-off

  • Platt/Isotonic на OOF; декомпозиция экономики → порог под целевые PD/долю одобрений.
  • Таблица decile LR до/после cut-off.

4) Стабильность

  • PSI по скору и топ-фичам (train vs OOT).
  • Slice-оценки (канал, регион, возраст).

5) Документация

  • Таргет/окна, сплиты; scale_pos_weight; калибровка; cut-off; PSI/KS/лифт; риски.

(Опционально) RI-лайт: propensity-веса для accepted-only и сравнить OOT.

19) Что зафиксировать в отчёте

  • Постановка PD@H, окна, правила default.
  • CV: Group/Time/Purged, OOT-период.
  • Дисбаланс: bad-rate, веса/ресемплинг.
  • Модель и фич-ядро (без утечек).
  • AUC/KS/PR-AUC (dev, OOT), калибровка (Brier/ECE), decile-backtest.
  • Cut-off и экономика; PSI/slice-метрики; (если было) RI-сценарий.
  • Reason-codes/монотонности и ограничения.
