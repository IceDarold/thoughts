Модуль 6. Keyword Spotting (KWS) под метрику harmonic(1−FRR,1−FAR)

Модуль 6 — Аудио KWS: модель + порог под метрику harmonic(1−FRR, 1−FAR) на кейсе «Не слышу тебя» (All Cups, Всерос-2025).

1) Результаты модуля (что должен уметь студент)

  • Корректно воспроизвести постановку: бинарная классификация аудиоклипа — есть ли в записи фразы «не слышу/не слышно» (1/0), данные в Opus, формат ответа id,label.
  • Считать локально целевую метрику:

    score = harmonic_mean(1−FRR, 1−FAR), где FRR=FN/NUM_POS, FAR=FP/NUM_NEG.

  • Построить надёжный пайплайн: декод Opus → препроцесс → признаки (mel/лог-mel) → CNN-базелайн → калибровка порога под целевую метрику (не accuracy/ROC-AUC).
  • Реализовать sliding-window инференс по длинным записям, агрегацию по окну/клипу и стабильный пост-процесс (решение 1/0).
  • Сдать валидный сабмит submition.csv (строго по образцу).

2) Теория (90–120 минут)

A. Постановка, данные, формат

  • Задача: определить наличие хотя бы одной целевой фразы («не слышу»/«не слышно») в аудиозаписи звонка. Метка 1 — фразы есть; 0 — нет. Файлы даны пакетами train_data.tar/test_data.tar.
  • Структура train: audio/*.opus + word_bounds.json со временем начала/конца целевой фразы (если отсутствует — запись без фразы). Это позволяет строить n-секундные положительные окна вокруг меток и отрицательные окна вне меток.
  • Формат сабмита: submition.csv с колонками id,label, где id — имя файла (без расширения), label∈{0,1}.

B. Метрика и её импликации

  • Harmonic(1−FRR,1−FAR) максимизирует баланс TPR (=1−FRR) и TNR (=1−FAR).

    Выводы:

    1. обязательна калибровка порога на валидации;
    1. полезен cost-sensitive подход/веса классов;
    1. важно контролировать ложные срабатывания (шумы/фоновые междометия) и пропуски (тихая речь).

C. Сигнальная цепочка и препроцесс

  • Декод Opus → mono → фиксированный sr (часто 16 kHz) — унификация перед признаками. Формат источника — Opus.
  • Нормализация амплитуды, удаление DC, опционально VAD (если даёт выигрыш по времени).
  • Оконное представление (sliding window): win=0.96–1.5 c, hop=0.25–0.5 c — баланс детальности/скорости.

D. Признаки (feature engineering)

  • Log-Mel спектр (n_mels=64/80, fmin=60–100 Гц, fmax=sr/2), иногда MFCC как дополнительный канал.
  • Аугментации: time-shift, pink/white noise (SNR 5–20 dB), time-stretch (±5–10%), SpecAugment (frequency/time masks).
  • Баланс классов: oversample позитивные окна; hard negative mining (ложные срабатывания из вал-окна).

E. Разметка обучающих примеров (используя word_bounds.json)

  • Позитивы: окна, перекрывающие [start,end] фразы; Негативы: окна в оставшейся части клипа (без пересечения).
  • Важно: никаких меток из test; валидация на чистых id, без утечек во времени/окнах.

F. Модели

  • Бейзлайн-CNN: 3–5 блоков Conv-BN-ReLU-Pool на log-mel, затем Global Max/Avg Pool → FC.
  • Более сильные: CRNN (Conv + BiLSTM/GRU), или компактные AST-варианты/конформеры (если железо/веса доступны).
  • Калибровка: temperature scaling/Platt по валидации — помогает порогу.

G. Валидация и выбор порога

  • CV по записям: KFold/Stratified по id, а не по окнам (иначе утечка).
  • Для каждого фолда: снимаем кривую score(τ) по τ∈[0,1] на вал-клипах (агрегируя оконные постериоры) → берём τ*, максимизирующий метрику.
  • Итоговый τ_ens: медиана/среднее по фолдам; при сабмите можно подвинуть τ в ±ε, если hold-out это показывает.

H. Инференс по целому клипу

  • Считаем постериор p(window) для каждого окна → агрегируем в клип:

    p(clip)=max_pool (чувствителен к единичным пикам) или top-k mean (устойчивее к шуму).

  • Метка: 1[p(clip)≥τ]. Логируем FRR/FAR по валидации, контролируем дисбаланс.

I. Скорость и I/O

  • Батчи окон, кэш мел-спектров, распараллеливание декода Opus → WAV/PCM.
  • Экономия памяти: хранить мел-тензоры в float16 при инференсе.

3) Практика (3–4 часа, лабы)

Лаба 1. Декод Opus, EDA, локальная метрика

  1. Распаковать train_data.tar: прочитать список audio/*.opus, word_bounds.json.
  1. Написать metrics_kws.py:

    confusion(y_true,y_pred) → (TP,FP,FN,TN);

    FRR=FN/NUM_POS, FAR=FP/NUM_NEG;

    score=harmonic_mean(1−FRR,1−FAR); проверить на игрушечных наборах.

  1. Быстрый EDA: длины клипов, доля позитивов/негативов, распределение длительности фразы.

Артефакты: decode_opus.py, metrics_kws.py, eda_kws.ipynb.

Лаба 2. Сбор датасета окон (windows) без утечек

  1. Из word_bounds.json построить позитивные окна (перекрывающие [start,end]) и негативные окна (вне интервала).
  1. Сохранить индексы окон (start_sec, end_sec), раздельно train/val по id.
  1. Сгенерировать log-mel для каждого окна, добавить аугментации.

Артефакты: build_windows.py, features_mel.py, кэш mel_cache/.

Лаба 3. CNN-базелайн и первый порог

  1. Архитектура: Conv×4 → GlobalPool → FC; loss — BCE, class weights по дисбалансу.
  1. Валидация по клипам: оконные постериоры → p(clip)=max_pool.
  1. Снять score(τ) на вал-клипах, выбрать τ*, зафиксировать baseline.

Артефакты: train_cnn.py, predict_windows.py, val_threshold.py.

Лаба 4. Усиление: аугментации, CRNN/другие аггрегаторы

  1. Добавить SpecAugment (time/freq mask), time-stretch, SNR-noise.
  1. Альтернатива: CRNN (Conv → BiGRU → Pool) или top-k mean агрегатор вместо max.
  1. Пересчитать метрику, сравнить с baseline.

Артефакты: augment.py, train_crnn.py, ablation_aug.md.

Лаба 5. Инференс на тесте и сабмит

  1. Для каждого test/*.opus — sliding window → p(window) → агрегация → p(clip).
  1. Брать τ_ens (медиана по фолдам) → label=1[p(clip)≥τ_ens].
  1. Собрать submition.csv в точном формате (id,label), проверить валидатором.

Артефакты: infer_clip.py, make_submission.py, submition.csv.

4) Критерии сдачи

  • Есть корректная локальная реализация метрики harmonic(1−FRR,1−FAR) и отчёт по FRR/FAR на валидации.
  • Чёткий протокол без утечек (разделение по id; fit на train, валидация на hold-out).
  • Улучшение baseline-score за счёт аугментаций/агрегаторов; фикс-seed и воспроизводимость.
  • Валидный сабмит submition.csv с нужным числом строк и правильными id.
