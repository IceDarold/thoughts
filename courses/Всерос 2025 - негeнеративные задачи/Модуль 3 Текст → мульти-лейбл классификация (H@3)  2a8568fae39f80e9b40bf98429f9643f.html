Модуль 3. Текст → мульти-лейбл классификация (H@3): «Мыслим категориями»

Модуль 3 делаем под мульти-лейбл текстовую классификацию с метрикой H@3 на кейсе «Мыслим категориями» (All Cups). Формат: сначала теория, затем пошаговые лабы до валидного сабмита.

1) Результаты модуля

  • Понимать постановку задачи: по app_name, shortDescription, full_description предсказать до трёх категорий в порядке убывания уверенности. Формат ответа — .tsv, ориентируемся на sample_submission.tsv.
  • Чётко считать Hitrate@3 (H@3) локально ровно как на площадке (попадание есть, если хотя бы одна правильная метка вошла в топ-3).
  • Собрать надёжный пайплайн: CV с итеративной стратификацией, линейные базелайны (TF-IDF + логрег/SGD), эмбеддинговый ранжировщик (sentence-эмбеддинги), лёгкий трансформер-классификатор; пост-процессинг «всегда ровно 3 метки».
  • Уметь блендить подходы и документировать анти-утечки и проверку формата.

2) Теория (90–120 минут)

A. Постановка и данные

  • Цель: по app_name, shortDescription, full_description предсказать до 3 категорий (упорядоченно). Обучающие данные содержат labels_str (истинные категории через |), тест без меток. В материалах есть sample_submission.tsv, train.tsv, test.tsv.
  • Метрика: H@3 — доля объектов, у которых хотя бы одна истина попала в топ-3 предсказаний. Формальное описание и примеры — в постановке задачи.

B. H@3: свойства и следствия для модели

  • Точность порядка важна меньше, чем наличие хотя бы одной релевантной метки сверху; значит при бленде лучше повышать recall в топ-3, даже ценой легкой потери ранжирования внутри тройки.
  • Выбор ровно 3 меток обязателен — проектируем пост-проц, который всегда возвращает три (в т.ч. при низких уверенности/пустом тексте).

C. CV и анти-утечки для мульти-лейбла

  • Итеративная стратификация по меткам (многоярлыковая) vs обычный KFold — цель: стабильные доли частых/редких классов в фолдах.
  • Анти-утечки: не допускать «подглядывания» в тест при расчёте любых статистик (словари, priors), любые трансформеры/векторизаторы — только fit на train с OOF протоколом.

D. Представления текста

  • Быстрые признаки: TF-IDF (word/char n-gram), двуязычный шум, нормализация Unicode, lower-casing, очистка HTML/URL.
  • Эмбеддинги предложений (sentence transformers): косинусная близость к прототипам классов (средние эмбеддинги текстов класса).
  • Аугментации (бережно): drop-words/typo-noise; контроль, чтобы не ломать домен.

E. Модели

  • Линейные: One-vs-Rest Logistic Regression / Linear SVM / SGDClassifier → быстрый сильный бейзлайн на TF-IDF.
  • Эмбеддинговый ранжировщик: nearest-prototype / kNN по косинусу; даёт устойчивый recall@3.
  • Трансформер-классификатор: компактный м-BERT/Distil-mBERT, loss — BCEWithLogits (multi-label), на инференсе берём топ-3 логитов.

F. Классовые дисбалансы и калибровки

  • Веса классов в логлоссе; temperature scaling/Platt для выравнивания вероятностей; priors по частотам как бэк-офф при пустых/слабых текстах.

G. Пост-процессинг (инварианты сабмита)

  • Ровно три метки: (1) слияние ранков из нескольких моделей → (2) tie-break по priors/PMI → (3) гарантированное добавление частотной метки при пустом топе.
  • Валидация формата .tsv и сортировка по убыванию уверенности. Ориентир — sample_submission.tsv.

3) Практика (3–4 часа, лабы)

Лаба 1. Реплика постановки и локальная H@3

  1. Разобрать train.tsv, test.tsv, посмотреть поля app_name, shortDescription, full_description, labels_str.
  1. Реализовать metrics.py::hitrate_at_3(y_true_sets, y_pred_top3) в точном соответствии с описанием задачи. Тесты на кейсах «1 истина», «2 истины», «нет совпадений».
  1. Собрать sample_submission.tsvмок в верном формате (пустые метки заменять на три самые частые — временно).

Артефакты: metrics.py, eda_text.ipynb, make_sample_submit.py.

Лаба 2. TF-IDF → логрег (скоростной бейзлайн)

  1. Препроцесс: чистка, токенизация, word/char n-gram, ограничение словаря, OOF-протокол.
  1. Модель: OvR Logistic Regression/SGD, оптимизация C/alpha, class-weights.
  1. Инференс: объединить тексты (name + short + full), получить вероятности по всем классам → выбрать топ-3.
  1. Оценить офлайн H@3; сохранить сабмит .tsv с тремя метками.

Артефакты: tfidf_vectorizer.pkl, train_lr.py, predict_lr.py, submit_lr.tsv.

Лаба 3. Эмбеддинговый ранжировщик (sentence-level)

  1. Получить sentence-эмбеддинги для текстов train; посчитать прототипы классов (mean embedding по примерам метки).
  1. Для каждого тестового текста — косинус до всех прототипов → ранжирование → топ-3.
  1. Сравнить H@3 с линейным базелайном; затем сделать ранговый бленд (сумма нормированных рангов/скор).

Артефакты: embeddings_cache/, ranker_embeddings.py, submit_emb.tsv.

Лаба 4. Трансформер-классификатор (лёгкий)

  1. Модель mBERT/Distil-mBERT, head для multi-label (BCEWithLogits).
  1. CV с итеративной стратификацией; early-stopping по вал-H@3 (через «топ-3 логитов»).
  1. Инференс: объединить логиты фолдов (OOF для анализа), взять топ-3 метки.

Артефакты: train_bert.py, predict_bert.py, submit_bert.tsv.

Лаба 5. Пост-проц и финальный бленд

  1. Инварианты: всегда 3 метки, устранение дублей/неизвестных классов, упорядочивание.
  1. Бленд: взвешенная сумма нормированных скор (LR/Emb/BERT) + priors; грида по весам с валидацией H@3.
  1. Финальная проверка формата: .tsv как в sample_submission.tsv.

Артефакты: blend.py, postprocess.py, final_submit.tsv.

4) Критерии сдачи модуля

  • Репозиторий: metrics.py (H@3), cv.py (итеративная стратификация), train_*.py, predict_*.py, blend.py, postprocess.py, README.md с командами.
  • Валидный сабмит .tsv, ровно три метки на строку, порядок убывания уверенности, проверен скриптом.
  • Офлайн H@3 устойчив на разных сплитах; отчёт по абляциям (TF-IDF vs Emb vs BERT, вклад пост-проц).

  1. Реализовать fallback для пустых/очень коротких описаний (priors/частотные метки) и посчитать, как это влияет на H@3.
